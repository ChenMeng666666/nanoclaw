# 记忆系统审查与优化路线图（Phase + Task，P0-P3）

## 1. 目标与范围

本文件用于审查当前项目记忆系统的实现质量，并给出可执行的分阶段优化方案。  
目标是把现有实现收敛到“可持续、可验证、可扩展”的统一架构：

- 统一记忆写入/检索主链路，避免双轨实现割裂
- 对齐分层记忆（L1/L2/L3）与会话/用户/智能体等作用域治理
- 提升检索质量（关键词 + 向量 + 重排）与工程稳定性
- 完成从“功能可用”到“生产可治理”的演进路线

## 2. 对标原则（用于评估）

结合你要求的方向，本次按以下原则对照：

- **OpenClaw v2026.3.7 / v2026.3.8 的演进理念**：统一主链路、可观测、可持续迭代
- **QMD 思路**：混合检索（Lexical + Vector + Rerank）、上下文树化/分层组织、面向代理的可消费结果
- **mem0 思路**：多层记忆（会话/用户/代理）、记忆治理（添加/更新/删除）、长期记忆质量与召回优先级控制

> 说明：本次以仓库当前实现为审查对象，不进行代码开发，仅输出问题清单和开发拆解。

## 3. 当前实现快照（事实）

### 3.1 组件现状

- `MemoryManager`：L1 缓存 + DB 写入 + 简单向量检索  
  文件：`src/memory-manager.ts`
- `DefaultContextEngine`：实现了 ingest / assemble（BM25+Vector+RRF）/ afterTurn  
  文件：`src/context-engine/default-engine.ts`
- `ContextEngineRegistry`：支持引擎工厂与全局配置  
  文件：`src/context-engine/registry.ts`
- Runtime API 提供 `/api/memory/add|search|list`  
  文件：`src/runtime-api.ts`
- 记忆存储表 `memories`（SQLite）  
  文件：`src/db.ts`, `src/db-agents.ts`
- 定时反思与固化（主要 L2→L3）  
  文件：`src/reflection-scheduler.ts`

### 3.2 高风险事实

1. **双轨架构割裂**  
   - `MemoryManager` 与 `DefaultContextEngine` 并存，但主业务链路并未统一到一个 canonical pipeline。

2. **ContextEngine 主流程疑似未真正接入**
   - `index.ts` 中 `contextEngines` 仅声明与遍历，未发现实例创建与注入主消息处理链路。
   - `createDefaultContextEngine(...)` 在 `index.ts` 内未被调用。

3. **DefaultContextEngine 存在占位实现**
   - `getRecentMessages(...)` 直接返回空数组，导致 `assemble(...)` 的查询上下文可能退化。

4. **迁移逻辑实现错误**
   - `migrateMemory(...)` 更新内容与 importance，但未更新 `level`，与“L1→L2→L3 迁移”目标不一致。

5. **迁移调度断链**
   - `MemoryManager.migrateMemories()` 定义存在，但未被主流程调用；主循环中的迁移定时器是空实现。

6. **文档与实现不一致**
   - 文档写“访问 3 次迁移”，代码实际为“访问≥3 且 7 天未访问”。

7. **Memory 元数据定义与落库不一致**
   - `Memory` 类型有 `messageType/tags/sourceType/sessionId`，但 `memories` 表与 `createMemory` 未持久化这些字段。

8. **API 治理不足**
   - `/api/memory/*` 对 `level`、`limit`、`content` 长度、body size 缺少严格约束。

9. **检索扩展性不足**
   - 记忆检索主要是全量拉取 + 内存计算余弦排序，缺少 ANN、阈值、分段分页与质量回写策略。

10. **测试覆盖不足**
    - 核心记忆链路缺少系统化单测/集成测；`tests/*.ts` 不在 vitest 默认 include 中。

## 4. 问题分级（按优先级）

### P0（立即处理）

- 双轨架构导致行为不可预测（同一能力两套实现）
- ContextEngine 未接入主链路
- 迁移逻辑不改 `level` 的实现错误
- 迁移调度与文档承诺不一致
- API 输入缺乏关键防护（边界值、枚举、体积限制）

### P1（短期处理）

- 元数据模型未持久化（定义与存储断裂）
- 检索路径缺少统一融合与质量反馈闭环
- 记忆召回优先级与作用域（session/user/agent/global）策略不完整
- 关键测试矩阵缺失（回归风险高）

### P2（中期处理）

- 检索性能优化（ANN/预过滤/分页/冷热分层）
- 记忆治理策略（衰减、强化、冲突消解、去重升级）
- 可观测性指标体系（召回率、命中率、质量评分）

### P3（持续演进）

- 跨代理/组织记忆共享策略
- 自动化记忆质量评估与策略自适应
- 离线评测基准与长期 A/B 机制

## 5. 目标架构（建议）

## 5.1 统一主链路（Single Source of Truth）

- **写入统一入口**：消息处理 -> 标准化提取 -> 记忆分类 -> 存储
- **检索统一入口**：查询理解 -> 多路检索（Lexical/Vector）-> 融合重排 -> 作用域合并 -> 回写访问信号
- **迁移统一入口**：定时 + 事件触发双机制（阈值、时间衰减、重要性）

## 5.2 分层 + 作用域双维度

- 分层：L1（工作态）/L2（短期）/L3（长期）
- 作用域：Session/User/Agent/Global
- 查询时按策略聚合：优先 User + Session，再补 Agent + Global

## 5.3 数据模型统一

- 类型定义、数据库 schema、序列化与 API 契约必须一致
- 元数据（messageType/tags/sessionId/sourceType/timestampWeight）可落库、可检索、可过滤

## 5.4 检索质量闭环

- 混合召回 + RRF + rerank
- 检索后回写（访问计数、命中反馈、质量标记）
- 质量评估指标驱动策略调优

## 6. 开发 Phase 规划

## Phase 0：止血与对齐（架构纠偏）

- **目标**：修复当前“实现与承诺不一致”的硬伤，建立统一迁移方向
- **优先级**：P0

Tasks:

1. **T0-1（P0）确定唯一主链路与退役计划**
   - 明确 `MemoryManager` 与 `DefaultContextEngine` 的角色边界（建议：ContextEngine 为检索/组装主路径，MemoryManager 退化为基础服务层或并入 Engine）
   - 输出 ADR（Architecture Decision Record）

2. **T0-2（P0）修复迁移实现语义错误**
   - `migrateMemory` 必须正确更新 `level`
   - 明确迁移时 content 是否应加前缀，避免污染原语义

3. **T0-3（P0）恢复真实迁移调度**
   - 主循环启用真实迁移任务，打通调用链与监控日志

4. **T0-4（P0）对齐文档与代码规则**
   - 迁移条件单一真相来源（配置化），禁止文档与实现分离

5. **T0-5（P0）补齐 API 输入防护**
   - `level` 白名单、`limit` 上下界、`content` 长度阈值、body size 限制、错误码标准化

验收标准：

- 迁移行为可被测试稳定复现
- 文档与运行时规则一致
- memory API 的非法输入全部被拒绝并可观测

---

## Phase 1：模型统一与主链路重建

- **目标**：完成“类型 -> 存储 -> API -> 引擎”的一致性重构
- **优先级**：P0-P1

Tasks:

1. **T1-1（P0）打通 ContextEngine 主流程接入**
   - 消息 ingest、assemble、afterTurn 在真实对话路径中被调用
   - 清理未使用的 `contextEngines` 逻辑死角

2. **T1-2（P1）完善 `getRecentMessages` 数据来源**
   - 从真实消息存储读取，支持 chatJid + 时间窗口 + 限制条数

3. **T1-3（P1）统一 Memory 元数据落库**
   - schema migration 增加 metadata 列或 JSON 字段
   - `createMemory/getMemories/updateMemory` 全链路支持

4. **T1-4（P1）作用域策略落地**
   - 明确 session/user/agent/global 召回优先级
   - API 支持 scope/filter 控制

5. **T1-5（P1）检索回写机制**
   - 搜索命中后更新访问热度/最后访问时间，形成迁移依据

验收标准：

- 新旧链路行为一致性可验证，且主链路只保留一种
- metadata 可完整写入、查询、过滤
- 作用域召回策略可配置、可测试

---

## Phase 2：检索质量与记忆治理升级

- **目标**：把“可检索”提升为“高质量可检索”
- **优先级**：P1-P2

Tasks:

1. **T2-1（P1）混合检索统一编排**
   - BM25 + Vector + RRF + rerank 统一到标准 pipeline

2. **T2-2（P1）查询理解增强**
   - 查询扩展、意图归一、同义词与关键词提取策略可配置

3. **T2-3（P2）记忆生命周期治理**
   - 衰减、强化、冲突合并、去重升级（语义去重而非仅 hash）

4. **T2-4（P2）质量评分字段**
   - 为记忆条目引入质量/可靠性评分，参与检索排序与迁移

5. **T2-5（P2）召回解释能力**
   - 输出“为何命中”线索（分数、标签、作用域、时间权重）

验收标准：

- Top-K 相关性提升（离线基准可量化）
- 重复/冲突记忆明显下降
- 记忆命中具备可解释信息

---

## Phase 3：性能与规模化

- **目标**：确保长期运行成本与延迟可控
- **优先级**：P2

Tasks:

1. **T3-1（P2）向量检索性能优化**
   - 引入 ANN 或分段预过滤，避免全量扫描

2. **T3-2（P2）冷热数据策略**
   - L2/L3 冷热分层、批量压缩、归档策略

3. **T3-3（P2）批处理与异步化**
   - 嵌入生成、重排与迁移队列化，削峰填谷

4. **T3-4（P2）限流与资源保护**
   - API 级限流、查询超时、并发控制

验收标准：

- 检索延迟与资源占用可稳定在目标范围
- 高并发下不出现显著退化或雪崩

---

## Phase 4：可观测、评测与发布治理

- **目标**：形成工程闭环，避免再次“文档正确、实现偏离”
- **优先级**：P1-P3

Tasks:

1. **T4-1（P1）测试矩阵补齐**
   - 单测：迁移、检索、作用域、去重、API 校验
   - 集成测：真实消息流 -> 记忆写入 -> 检索 -> 迁移
   - E2E：跨会话与多用户隔离

2. **T4-2（P1）将关键测试纳入默认测试入口**
   - 避免 `tests/*.ts` 与 CI 主链路脱钩

3. **T4-3（P2）核心指标仪表板**
   - 召回率、命中率、误召回率、迁移成功率、检索耗时、缓存命中率

4. **T4-4（P3）灰度与回滚机制**
   - 为新检索策略与迁移规则提供可切换开关和回滚路径

验收标准：

- 关键回归可自动发现
- 每次策略调整可量化评估收益与风险
- 具备可控发布与回滚能力

## 8. 建议执行顺序（最短路径）

建议按以下顺序推进，以最小风险恢复记忆系统可信度：

1. `T0-1 -> T0-2 -> T0-3 -> T0-5`
2. `T1-1 -> T1-2 -> T1-3 -> T1-4 -> T1-5`
3. `T4-1 -> T4-2`（尽早纳入，防止回归）
4. 再进入 `T2` 与 `T3` 的质量/性能增强

## 9. 结论

当前系统不是“没有能力”，而是“能力分散且主链路未收敛”。  
优先完成 P0/P1 架构收敛后，再做质量和性能迭代，才能真正实现稳定、可持续的长期记忆体系。
