# NanoClaw 全局架构大纲

## 1. 文档定位

本文是当前蓝图的全局架构大纲，用于统一项目后续规划、实现、验证和演进时的上位约束。

在蓝图未完成前，所有方案评估、架构取舍与模块设计，都应优先服从本文，而不是局部实现便利。

本文吸收并提纯了阶段 0 的全部规划结论，作为后续阶段的统一入口。

## 2. 蓝图总目标

项目长期目标由五个阶段构成：

1. **阶段 0：蓝图治理与边界统一**
2. **阶段 1：Agent 独立运行单元**
3. **阶段 2：Agent 私有记忆系统**
4. **阶段 3：共同进化系统**
5. **阶段 4：Private Moltbook**
6. **阶段 5：向上知识反哺**

整体推进顺序固定为：

**先确定主体，再建立记忆，再形成进化，再扩展社会，再反哺母体。**

## 3. 架构总原则

### 3.1 主体优先

- agent 是最小能力主体
- 所有记忆、任务、反思、经验、社交行为都必须能追溯到明确 agent

### 3.2 上下文不等于主体

- group 是协作上下文，不是主体
- container 是运行壳，不是主体
- channel 是通信接口，不是主体

### 3.3 治理与运行分离

- 运行平面负责日常对话、任务执行、局部材料产生
- 治理平面负责建设、修复、审核、全局配置和知识上行筛选
- 治理平面不直接进入组内参与日常对话治理

### 3.4 稳定骨架与变化能力分离

- core 只承载稳定骨架
- custom module 承担长期结构定制
- skill 承担变化快、可插拔、可按需启用的能力

### 3.5 数据与状态隔离

- 源码、治理文档、组本地上下文、运行态状态必须分域
- 运行态数据不得混入源码域

### 3.6 高层知识逐级升级

- 原始材料不能直接上行为主程序规则
- 升级越往高层，验证越严格，数量越少

## 4. 对象模型

## 4.1 核心对象

### agent

agent 是系统中的最小能力主体，是记忆归属、任务执行、反思产出、经验积累、社交行为与知识上行的第一责任单元。

### group

group 是协作上下文与隔离单元，用于承载共享视图、局部权限边界和会话上下文。group 不是主体。

### main

main 是最上层建设者、修复者与审核入口，负责高权限治理、全局配置、异常修复与知识上行审核。main 不参与组内日常对话治理。

### container

container 是 agent 的隔离运行载体。container 可重建、迁移、销毁，因此不能作为记忆或经验的归属主键。

### channel

channel 是外部通信接口，负责消息接入、身份映射与消息送达，不负责认知归属与治理权。

## 4.2 认知对象

### memory

可在未来继续复用的信息单元，面向事实、偏好、约束、关系与上下文。

### reflection

对任务、失败、决策或修复过程形成的结构化复盘单元，强调原因分析。

### experience

从反思、日志和历史成败中提炼出的可迁移方法单元。

### knowledge

比 experience 更稳定、更少场景依赖的长期知识单元。

### upward knowledge

经筛选与审核后，可反哺更高系统层的知识产物。

### archive summary

对长日志、长记忆、长反思进行高密度压缩后形成的摘要对象，是后续高频读取入口之一。

## 4.3 默认判断

- agent 是最小主体
- group 是上下文，不是主体
- main 是治理入口，不是组内对话治理者
- 一个 group 未来允许承载多个 agent
- upward knowledge 必须通过 main 所代表的治理平面审核

## 5. 分层边界

## 5.1 架构层级表

| 层级 | 主职责 | 是否稳定长期存在 | 是否允许高频变化 | 是否直接拥有治理权 | 典型落位 |
|---|---|---|---|---|---|
| core | 编排、安全、调度、基础路由、最小全局机制 | 是 | 否 | 是 | `src/` 核心稳定文件 |
| custom module | 长期结构化定制能力 | 是 | 低到中 | 受 policy 约束 | 独立于 core 的定制目录 |
| skill | 外部交互、后台流程、专项能力 | 否，按需存在 | 是 | 否 | `.claude/skills/` |
| group local state | group 或 agent 的局部状态 | 否 | 是 | 否 | `groups/`、局部数据目录 |

## 5.2 分层准入规则

### 进入 core 的条件

- 属于所有 agent 或所有运行流程都必须依赖的骨架能力
- 与安全边界、基础调度、核心路由或宿主校验直接相关
- 不适合作为 skill 按需加载

### 进入 custom module 的条件

- 具有长期存在价值
- 不适合做成一次性 skill
- 不应侵入上游 core
- 能为记忆、进化、社会层提供结构支撑

### 进入 skill 的条件

- 外部通道接入
- 复杂后台任务
- 反思、归档、社会行为等专项机制
- 可替换、可独立发布、可按需启用的能力

### 停留在局部状态层的条件

- 某个 group 的短期上下文材料
- 某个 agent 的局部实验配置
- 仅对个别协作空间有效的共享文件
- 不具备全局复用价值的临时产物

## 5.3 禁入项

- 把记忆处理、进化流程、社会化机制硬编码进主循环
- 让 skill 直接承担治理规则定义权
- 让局部状态伪装成系统公共结构
- 用 container 运行边界替代 agent 身份边界
- 因实现方便而把运行态状态写回源码目录

## 6. 身份、归属与权限

## 6.1 归属总则

- 第一归属锚点是 agent
- 第二归属锚点是 group
- main 是高权限治理入口，不是新主体层

## 6.2 身份归属表

| 对象 | 是主体吗 | 主要角色 | 可管理范围 | 可写范围 | 说明 |
|---|---|---|---|---|---|
| agent | 是 | 最小能力主体 | 自身任务、记忆、行为 | 受上下文约束的局部写入 | 所有认知归属的第一锚点 |
| group | 否 | 协作上下文 | 本 group 共享视图 | 本 group 局部共享域 | 不是主体 |
| main | 否 | 建设、修复与审核入口 | 修复、审核、全局配置、异常处理 | 全局记忆、高权配置、治理入口 | 不直接进入组内日常治理 |
| container | 否 | 运行载体 | 运行期可见目录 | 挂载授权范围 | 不拥有认知归属 |
| channel | 否 | 外部通信接口 | 消息接入与发送 | 不应直接承担核心知识写入 | 负责映射，不负责治理 |
| skill | 否 | 扩展能力单元 | 自身执行范围 | 被授予的最小权限范围 | 不天然提升权限 |

## 6.3 权限原则

- 权限由宿主侧边界、目录可见面和写权限决定
- 权限不是提示词承诺，而是挂载与校验结果
- non-main 默认按不可信上下文处理
- main 是唯一默认高权限建设、修复与审核入口

## 6.4 跨边界规则

### main 允许的动作

- 管理全局记忆
- 管理注册 group
- 为任意 group 调度修复性或治理性任务
- 发起跨 group 的建设、修复与审核动作
- 审核 upward knowledge

### main 不应承担的职责

- 不直接进入 agent 所在组参与日常对话治理
- 不替代组内 agent 做常规协作决策
- 不把高权限误用成组内会话接管权

### non-main 禁止的动作

- 不得直接为其他 group 发送消息
- 不得直接为其他 group 调度任务
- 不得写入全局记忆
- 不得直接触发主程序规则变更

## 7. 数据边界

## 7.1 数据域分类

| 数据域 | 典型内容 | 是否进版本控制 | 说明 |
|---|---|---|---|
| 源码域 | `src/`、`container/`、稳定配置 | 是 | 不承载运行态状态 |
| 治理域 | `docs/ARCHITECTURE.md`、`TODO.md`、规划文档 | 是 | 用于规则、规划、决策沉淀 |
| 组本地域 | `groups/{name}/` 的记忆与文件 | 仅少量模板受控 | 用于共享上下文与局部材料 |
| 运行态状态域 | `data/`、`store/`、`logs/` | 否 | 用于 session、IPC、认证态、日志等 |

## 7.2 数据归位规则

- global memory 属于高权共享层，只能通过受控治理入口处理
- group memory 属于协作共享层，只对本 group 上下文可见
- agent 私有记忆应逐步脱离对 group 目录的完全依赖
- reflection 属于中间治理材料，不直接等同于 group 会话资产
- upward knowledge 属于治理资产，不属于组内会话资产

## 7.3 数据禁令

- 不允许把数据库、认证态、日志写进源码域
- 不允许把原始会话直接当作长期知识层
- 不允许让 group 文件夹承担全部未来运行态数据
- 不允许让上行知识直接来自未经筛选的原始日志

## 8. 记忆与进化生命周期

## 8.1 主链路

系统统一采用以下生命周期主链路：

**task / session / log / social interaction → memory → reflection → experience → knowledge → upward knowledge**

与之并行的成本控制链路为：

**长日志 / 长记忆 / 长反思 / 长经验 → archive summary**

## 8.2 各层职责

| 层级 | 解决的问题 | 不解决的问题 |
|---|---|---|
| 原始材料层 | 保留事实现场，提供后续输入 | 不直接上行，不直接充当知识 |
| memory | 以后还要不要记住 | 不解释原因，不总结方法 |
| reflection | 为什么会这样 | 不等于经验，不直接上行 |
| experience | 以后怎么做更好 | 不要求完全稳定，不直接替代知识 |
| knowledge | 什么已经稳定成立 | 不保存原始日志，不自动进入主程序规则 |
| archive summary | 如何降低 token 成本 | 不制造新知识，只压缩材料 |
| upward knowledge | 什么值得反哺母体 | 不接收原始材料，不跳过审核 |

## 8.3 状态升级原则

- 写入门槛最低的是 memory
- 升级为 experience 必须有反思或对照材料支撑
- 升级为 knowledge 必须弱化场景依赖并具备长期适用性
- 进入 upward knowledge 必须已经稳定化并经过治理平面审核

## 9. 触发器体系

## 9.1 写入触发器

- 用户显式要求记住
- 出现未来高概率复用的事实、偏好、约束
- 关系信息对后续任务持续有价值
- group 形成稳定共享上下文

## 9.2 反思触发器

- 解决复杂 Bug
- 失败后完成修复
- 完成阶段性任务
- 做出关键架构决策
- 同类问题再次出现

## 9.3 经验提炼触发器

- 反思中出现可迁移方法
- 有可比较的成功与失败样本
- 某方法在不同任务中重复奏效
- 某坑点具备高复发风险

## 9.4 知识升级触发器

- 同类经验跨多个任务仍成立
- 结论已明显弱化场景依赖
- 可抽象成稳定原则或规则
- 经治理平面或人工审核认可

## 9.5 归档触发器

- 日志、记忆、反思文档超过 500 行
- 同类材料持续堆积导致检索成本过高
- 某对象低频但仍需保留
- 某对象已被更新版本替代但仍有历史价值

## 9.6 读取触发器

- 开始新的编码任务前
- 遇到类似 bug 或相似任务前
- 准备做重大架构决策前
- 准备写入新的高层知识前
- 准备上行前

## 9.7 上行触发器

只有同时满足以下条件，才允许生成 upward knowledge candidate：

- 已不是原始日志或原始会话
- 已经过经验或知识层稳定化
- 具备可迁移价值
- 对主程序或治理规则具有增益
- 已经过 main 所代表治理平面的审核流程

## 10. 摘要与归档政策

## 10.1 摘要优先原则

后续系统日常读取优先级应为：

1. archive summary
2. stabilized knowledge
3. validated experience
4. recent reflection
5. raw memory
6. raw logs / session transcript

## 10.2 摘要内容最低要求

- 核心结论
- 关键原因
- 关键关系
- 适用边界
- 禁止误用场景

## 10.3 归档后使用方式

- 主程序默认先读摘要
- 只有摘要不足以支撑判断时，才回看原始材料
- 归档不等于遗忘，而是把高频入口切换到压缩层

## 11. 阶段 0 验收结论

阶段 0 完成的判定标准如下：

- 核心术语已有唯一且稳定的定义
- 主体、数据、权限、上行边界已经明确
- 生命周期主链路与触发器已经明确
- 后续阶段 1 到阶段 5 都有可引用的治理前提
- 对什么能做、什么暂缓、什么不能做已有清晰判断

在当前架构下，阶段 0 已视为完成。

## 12. 阶段 1 入口

阶段 1 为 **Agent 独立运行单元**，目标是把“agent 作为独立主体”正式立住。

阶段 1 的重点拆解为三部分：

1. **身份模型**
2. **配置模型**
3. **运行模型**

阶段 1 的最低目标包括：

- agent 身份边界清晰
- agent 配置与 group 上下文解耦
- agent 拥有独立运行容器与独立模型配置能力
- 后续记忆、进化、社会能力都能绑定到明确 agent

## 13. 当前缺口

当前仍需在后续阶段逐步补齐的正式落地项包括：

- `ROADMAP.md`
- `docs/REFLECTIONS.md`
- agent 私有记忆的独立表达
- experience / knowledge / graph relation 的正式持久化分层
- 进化数据集与图谱数据的目录规范

## 14. 使用规则

- 后续新增方案必须先判断自己属于哪一阶段
- 若某方案跨多个阶段，优先服务更靠前的依赖阶段
- 若某设计破坏本文定义的前置边界，应视为暂缓项
- 若后续实现与本文冲突，应优先修正实现，不倒置为“现状定义蓝图”
