# 学习体系审计与整改路线图（容器内学习 / 记忆 / 进化）

## 1. 审计目标与范围

本次审计聚焦以下目标是否落地并一致：

1. 学习行为应发生在 **agent 容器内**，由 agent 参考 `agent-learning` skill 自主执行。  
2. 当用户要求“学习某方向”时，应先检查反思定时任务（每小时/每天/每周/每月/每年），不存在则创建，存在则进入下一步。  
3. 应按用户指定节奏创建学习任务（固定时间或每隔多久）。  
4. 学习闭环应与记忆系统、进化系统协同，并可持续迭代同步到容器内学习系统。

> 说明：本文件仅做审计与改造规划，不包含代码改动。

---

## 2. 当前实现现状（As-Is）

### 2.1 主进程与容器执行模型

- 容器是按消息/任务触发启动的短生命周期执行单元（`docker run --rm`），并非长期驻留学习容器。  
- 主进程启动后会常驻启动反思调度器（hourly/daily/weekly/monthly）并做记忆固化。

### 2.2 学习体系入口

- Runtime API 提供学习相关接口（计划创建、任务开始/完成、反思生成、每日总结等）。  
- `agent-learning` skill 提供了容器内脚本（初始化、学习自动化、反思 cron、每日计划）。  
- 学习任务完成后，已有“反思 -> 记忆（L3）-> 条件提交进化库”的链路。

### 2.3 用户“学习某方向”触发方式

- 当前主消息处理链路没有硬编码“学习意图 -> 学习工作流”的确定性编排。  
- 更接近“把用户消息交给 agent，自行决定是否调用学习 API / skill”。

## 3. 关键偏差与问题清单

## P0（必须优先修复）

- [x] **反思频率协议不一致（yearly 断裂）**  
  - 容器脚本支持 `yearly`，但 Runtime API 触发接口仅允许 `hourly|daily|weekly|monthly|task`。  
  - 结果：脚本侧 yearly 触发无法在 API 侧被一致处理，形成“配置宣称支持、后端实际不支持”的断层。

- [x] **学习计划自动排程类型与调度器协议不一致**  
  - 学习计划创建时写入 `schedule_type: daily|weekly|monthly`。  
  - 任务调度器计算下次执行仅支持 `cron|interval|once`。  
  - 结果：学习计划生成的定时任务可能只执行一次或行为不可预期，无法形成稳定周期学习。

- [x] **容器脚本与 API 认证策略不一致**  
  - Runtime API 默认要求 `X-API-Key`。  
  - 学习脚本默认调用 `curl` 未携带 `X-API-Key`。  
  - 结果：在默认安全配置下，容器内学习自动化调用会被拒绝。

## P1（高优先级）

- [ ] **“用户指定学习方向”缺少系统级确定性编排**  
  - 目前依赖 agent 自主理解并调用 skill/API，缺少平台层“识别 -> 校验/创建反思计划 -> 创建学习任务 -> 反馈”的强约束流程。  
  - 结果：行为一致性依赖模型当下表现，不满足“应当执行”的产品语义。

- [ ] **用户时间意图未真正落地到排程语义**  
  - 计划创建逻辑默认每天 `20:00`，按 phase 逐日安排。  
  - 用户的“每天固定时间/每隔多久”未映射为 `cron` 或 `interval` 的标准计划表达。  
  - 结果：调度无法体现用户时间偏好，学习体系“可配置性”与“可解释性”不足。

- [ ] **容器学习自动化脚本字段与接口契约存在错位风险**  
  - 脚本执行 daily plan 时，任务字段使用与 `/api/learning/task/start` 期望字段不完全一致。  
  - 结果：脚本自动执行链路存在失败概率，稳定性不足。

- [ ] **学习模型策略未分层治理（本地增强 vs 远端 SDK）**  
  - 当前“学习需求分析”存在可选本地 LLM 增强，反思链路主要是规则/模板逻辑；主 agent 执行走远端 SDK。  
  - 结果：模型能力层级不一致，复杂学习方向与高质量反思场景下稳定性和上限受限。

## P2（中优先级）

- [ ] **配置文件命名与读取路径不统一**  
  - skill 中存在 `learning-automation.json`、`skill-learning-schedule.json`，脚本侧默认读取 `automation.json`、`schedule.json`。  
  - 结果：默认配置可能不生效，行为依赖回退分支。

- [ ] **主进程反思调度与容器内 cron 调度职责重叠**  
  - 存在双轨反思调度（主进程 + 容器脚本），未定义单一事实来源（SSOT）。  
  - 结果：可能重复触发、排障复杂、运营口径不一致。

- [ ] **学习自动化状态仅为内存标记，缺少执行编排能力**  
  - `/api/learning/automation/start` 当前主要更新内存集合状态。  
  - 结果：状态与实际定时执行系统解耦，难以保障“start 即真正开始调度”。

## P3（治理与可维护性）

- [ ] **学习体系文档与真实运行语义存在漂移**  
  - 文档中“自动初始化、自动 cron、yearly 支持、完整闭环”部分与运行事实存在偏差。  
  - 结果：新成员理解成本上升，回归验证困难。

- [ ] **学习链路缺少端到端验收基线**  
  - 缺少“用户说学习方向 -> 定时任务检查/创建 -> 排程执行 -> 反思 -> 记忆 -> 进化提交”的自动化验收场景。  
  - 结果：后续迭代易回归。

## 4. 目标架构（To-Be）

## 4.1 架构原则

- [ ] **单一调度真相源（SSOT）**：统一由主进程调度学习与反思，容器内脚本作为执行助手而非第二调度中心。  
- [ ] **契约先行**：学习计划、反思频率、时间表达统一协议（推荐标准化为 `cron|interval|once` + reflection enum）。  
- [ ] **意图确定性编排**：对“学习某方向”提供平台侧可审计工作流，不依赖纯提示词偶然触发。  
- [ ] **安全默认开启**：容器内 API 调用统一走密钥注入与校验。  
- [ ] **闭环可验收**：每条链路有可重复的 E2E 验证。

## 4.2 推荐职责边界

- [ ] **主进程**：意图识别编排、任务排程、反思调度、记忆与进化入库。  
- [ ] **容器内 agent + skill**：学习执行、上下文推理、结果回传。  
- [ ] **Runtime API**：统一受控入口、参数校验与鉴权。  
- [ ] **数据层**：学习任务、调度任务、反思记录、学习结果、演化记录。

## 5. 模型分层基线（细化版，建议纳入实施基线）

## 5.1 分层目标

- [ ] **质量优先**：高价值学习决策与反思总结优先使用 SDK 远端模型。  
- [ ] **成本可控**：低风险检索增强、候选扩展允许本地模型。  
- [ ] **稳定可回退**：任一层失败都可降级，不阻断学习主流程。  
- [ ] **行为可审计**：每次模型选择、回退原因、质量分都可追踪。

## 5.2 链路分层规则（必须执行）

- [ ] **L0（硬规则层）**：参数校验、任务状态机、调度创建、鉴权检查，禁止依赖 LLM。  
- [ ] **L1（本地模型优先层）**：查询扩展、关键词重写、低风险候选生成，默认本地模型。  
- [ ] **L2（SDK 优先层）**：学习需求分析、学习计划结构化生成、关键反思草案，默认 SDK。  
- [ ] **L3（强约束层）**：进化提交前总结、高价值反思、跨周期学习策略，必须 SDK + 规则二次校验。

## 5.3 API 到模型策略映射（建议）

- [ ] **`/api/learning/analyze-needs`**：SDK 优先，本地仅作候选增强，不直接产出最终决策。  
- [ ] **`/api/learning/generate-daily-plan`**：SDK 负责计划生成，规则层负责时间窗合法性与冲突消解。  
- [ ] **`/api/learning/reflection/generate`**：分级处理；普通反思可规则模板，高价值反思走 SDK。  
- [ ] **`/api/learning/analyze-outcome` 与 `/api/learning/extract-knowledge`**：先规则提取，再按质量阈值选择是否追加 SDK 精炼。

## 5.4 回退与降级策略（必须具备）

- [ ] **回退顺序**：SDK 失败 -> 本地模型 -> 规则模板；每步都写入 `degrade_reason`。  
- [ ] **超时策略**：SDK 调用超时后进入降级路径，不允许阻塞学习任务状态更新。  
- [ ] **熔断策略**：连续失败达到阈值后短时熔断 SDK，自动切到本地/规则并告警。  
- [ ] **恢复策略**：熔断窗口结束后按探测流量恢复 SDK，恢复期间保留双写观测。

## 5.5 质量门禁与验收指标（首版）

- [ ] **计划可执行率**：生成计划可直接入库且调度合法的比例 >= 95%。  
- [ ] **反思有效率**：反思文本满足结构完整性与可行动项占比 >= 90%。  
- [ ] **降级触发率**：SDK 异常导致降级的比例可观测且有周报。  
- [ ] **人工复核通过率**：抽样复核中“方向正确+建议可执行”通过率持续提升。  
- [ ] **端到端时延**：学习请求到计划落库/反思落库时延有上限并可报警。

## 5.6 Qwen3.5-2B GGUF 使用边界

- [ ] **允许场景**：query expansion、候选关键词扩展、低风险归类。  
- [ ] **限制场景**：复杂学习需求分析、高价值反思、跨任务策略制定不作为最终模型。  
- [ ] **治理结论**：Qwen3.5-2B GGUF 作为“增强与兜底层”是合理的，作为“决策主层”不建议。

## 6. 开发 Phase 拆解与 Task 列表（P0-P3）

## Phase 0：契约对齐与止血（P0）

- [x] **[P0] 统一反思频率协议**：对齐 `yearly` 的端到端支持或明确废弃并同步脚本/接口/文档。  
- [x] **[P0] 统一调度类型协议**：学习计划创建必须输出调度器可识别的 `cron|interval|once`。  
- [x] **[P0] 打通 API 鉴权链路**：容器学习脚本调用 Runtime API 时统一携带密钥。  
- [x] **[P0] 增加契约回归测试**：覆盖 reflection type、schedule_type、认证头校验三类断裂点。
- [x] **[P0] 输出学习模型分层基线**：明确哪些链路允许本地模型，哪些链路必须走远端 SDK。

## Phase 1：学习意图确定性编排（P1）

- [ ] **[P1] 建立学习意图编排入口**：识别“学习方向”请求并进入标准学习工作流。  
- [ ] **[P1] 增加“先检查反思定时任务”步骤**：缺失即创建，存在则复用。  
- [ ] **[P1] 实现用户时间偏好解析**：将“每天固定时间/每隔多久”映射为标准调度表达。  
- [ ] **[P1] 学习计划创建链路标准化**：计划、任务、定时任务三者字段一一对齐并可追踪。
- [ ] **[P1] 升级学习需求分析模型策略**：将本地 LLM 增强切换为 SDK 优先、本地兜底的双层策略。  
- [ ] **[P1] 升级反思生成策略**：高价值反思采用 SDK 生成并保留规则兜底，沉淀可审计质量指标。

## Phase 2：容器学习系统与主系统协同（P2）

- [ ] **[P2] 明确双轨调度治理策略**：决定保留主调度还是容器 cron，并统一为单轨主责。  
- [ ] **[P2] 修复脚本与接口字段契约**：确保 daily plan 执行脚本可稳定调用 task start/complete。  
- [ ] **[P2] 统一配置文件命名与加载路径**：消除 `automation/schedule` 与 `learning-automation/skill-learning-schedule` 漂移。  
- [ ] **[P2] 打通学习自动化状态与真实执行态**：`start/stop/status` 与实际调度状态一致。

## Phase 3：治理、文档、验收（P3）

- [ ] **[P3] 更新学习体系架构文档**：按真实实现重写链路图、协议和职责边界。  
- [ ] **[P3] 补齐运维手册**：排障路径、观测指标、故障演练、回滚策略。  
- [ ] **[P3] 建立端到端验收矩阵**：用户学习指令 -> 调度 -> 执行 -> 反思 -> 记忆 -> 进化。  
- [ ] **[P3] 建立变更门禁**：学习体系契约变更需通过 API 契约测试和集成回归。

## 7. 建议的验收标准（按阶段）

- [x] **Phase 0 验收**：无 reflection/schedule/auth 协议断裂；关键 API 契约测试通过。  
- [ ] **Phase 1 验收**：用户发出学习方向后，系统可稳定产出可追踪学习计划与定时任务。  
- [ ] **Phase 2 验收**：容器脚本、主进程调度、Runtime API 三方协议一致且可重复执行。  
- [ ] **Phase 3 验收**：文档、监控、测试矩阵与实现一致，支持持续迭代。

---

## 8. 实施顺序建议

- [x] 先做 **P0 契约止血**，避免继续产生错误学习任务与不可执行调度。  
- [ ] 再做 **P1 编排确定性**，把“应当学习”从提示词行为升级为系统能力。  
- [ ] 随后做 **P2 协同收敛**，确保容器学习体系与主系统不会双轨漂移。  
- [ ] 最后做 **P3 治理固化**，把这套能力变成可维护、可审计、可回归的工程资产。
