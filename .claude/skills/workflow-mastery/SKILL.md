# Workflow Mastery v2

---

## 0. 入口判断（先读这里）

**紧急线上问题？** → 直接跳 [Fast Track](#fast-track)

**否则，任务在哪个阶段？**

| 当前状态 | 跳转 |
|----------|------|
| 需求模糊 / 没有规格 | 阶段1 |
| 有想法但没有计划 | 阶段2 |
| 有计划，开始执行 | 阶段3 |
| 代码写了一半 / 有 Bug | 阶段4 |
| 实现完成，准备收尾 | 阶段5 |
| 被纠正 / 出现偏差 | 阶段6 |

**回溯规则**：执行中发现计划有误 → 回到阶段2重新规划，不要强行往前走。

---

## Fast Track（紧急修复）

> 适用：线上故障、阻塞性 Bug、必须立刻修的问题

跳过阶段1-2，直接走：**阶段4（调试）→ 阶段5（评审）→ 提交**

事后补做阶段6（记录教训），不补做其他阶段。

---

## 阶段1：需求澄清

> 不要立刻写代码。先用最少的问题挖出真实需求。

一次最多问 **3个问题**，按优先级排序，等用户回答后再继续。

必问：
1. **目标**："你真正想解决的问题是什么？"
2. **约束**："有什么不能动的？（技术栈/时间/团队）"
3. **成功标准**："完成后你怎么判断它成功了？"

有了答案后，给出 **2种方案对比**（不要超过3种，减少决策疲劳）：

```markdown
## 方案对比

| | 方案A：[名称] | 方案B：[名称] |
|-|--------------|--------------|
| 实现复杂度 | 低 / 中 / 高 | 低 / 中 / 高 |
| 主要优势 | ... | ... |
| 主要风险 | ... | ... |
| 推荐场景 | ... | ... |

**推荐**：[方案X]，因为 [一句话理由]
```

用户确认后 → 保存到 `tasks/design.md`，进入阶段2。

---

## 阶段2：写计划

> 在任何代码之前，先用子智能体探索代码库，再写计划。

### 探索提示词模板
```
分析 [目标范围] 并汇报：
1. 相关文件结构与现有模式
2. 潜在破坏点
3. 推荐实现路径
不要修改任何文件。
```

### 计划格式（tasks/todo.md）

```markdown
## 任务 N：[标题]
- 文件：`src/exact/file.ts`
- 前置：任务 M
- 验证：`npm test src/exact/file.test.ts` → 期望输出 [X]
- 回滚：[见下方回滚分类]

### 测试（先写）
[测试代码]

### 实现（后写）
[实现代码]
```

**回滚分类**：
- 纯代码改动 → `git checkout <file>`
- 依赖变更 → `git checkout package.json && npm install`
- 数据库迁移 → 记录 down migration 命令
- 多文件 → `git stash` 或 `git revert <commit>`

**计划质量门槛**（不达标就重写）：
- ✅ 每任务 2-5 分钟粒度
- ✅ 精确文件路径，无"在某处修改"
- ✅ 可执行的验证命令
- ✅ YAGNI：只做需要的
- ❌ 禁止"等等"/"其他必要更改"

## 阶段3：执行

### 选策略

**有子智能体可用** → 策略A（并发）
**无子智能体 / 任务强依赖** → 策略B（批次）

**策略A：子智能体驱动**

每个任务派发独立子智能体，携带：任务规格 + 相关代码片段（不是整个代码库）+ 验证标准。

两阶段评审：
1. 规格合规：实现是否符合计划？
2. 代码质量：有无技术债/反模式？

**策略B：批次执行**
```
批次1（并行，无依赖） → 人工检查点 → 批次2 → ...
```

### 上下文管理

- 探索任务交给子智能体，结果只汇总关键发现到主上下文
- 主上下文只保留：当前任务 + lessons.md 摘要
- **上下文接近上限时**：暂停，输出当前进度快照到 `tasks/progress.md`，开启新会话并加载快照继续

## 阶段4：TDD + 系统调试

### TDD 循环
```
🔴 RED   → 写失败测试，先看它失败
🟢 GREEN → 写最少代码让测试通过
🔵 REFACTOR → 重构，保持绿色
```

铁律：测试前写的代码删掉重写。提交前所有测试必须绿色。

### 系统调试四阶段

| 阶段 | 核心问题 | 工具 |
|------|----------|------|
| **隔离** | 最小复现场景？哪一层？哪次变更引入？ | git bisect, 日志 |
| **根因** | 症状→直接原因→根本原因（5 Why） | 断点, 断言 |
| **防御** | 修根因，加防御性检查，写回归测试 | 测试覆盖率 |
| **验证** | 原始场景通过？边界覆盖？无副作用？ | 全量测试 |

**不猜测，用证据证明每一步假设。**

## 阶段5：评审 & 收尾

### 提交前检查（tasks/review.md）

```markdown
### 规格合规
- [ ] 所有计划任务完成
- [ ] 无计划外变更（有则说明原因）

### 代码质量
- [ ] DRY / YAGNI / 命名清晰 / 错误处理完整

### 测试
- [ ] 全绿 / 新功能有测试 / 边界覆盖

### 影响
- [ ] DB 迁移安全 / API 向后兼容 / 性能无明显劣化
```

### 问题分级

| 级别 | 描述 | 处理 |
|------|------|------|
| 🔴 Critical | 数据丢失/安全/功能破坏 | 阻塞，必须修复 |
| 🟡 Major | 性能/技术债/测试缺失 | 当前迭代修复 |
| 🟢 Minor | 风格/可读性 | 记录，下次迭代 |

收尾时询问用户：合并主分支 / 创建 PR / 保留分支 / 放弃变更。

## 阶段6：教训记录

> 被纠正后立即执行，Fast Track 结束后也要补做。

```markdown
## [YYYY-MM-DD] [简短标题]

- 发生了什么：[一句话]
- 根本原因：需求误解 / 跳过计划 / 忽略现有模式 / 测试不足 / 其他
- 下次怎么做：[具体行为，不是口号]
- 适用范围：[哪类任务]
- 有效期至：[留空 = 长期有效；填日期 = 到期归档]
```

**维护规则**：
- 每次新任务开始前，快速扫一遍 lessons.md 标题
- 每 10 条做一次元总结：提炼成 1-3 条原则，旧条目移入 `tasks/lessons-archive.md`
- 超过有效期的条目移入归档，不删除

## 文件结构

```
tasks/
├── design.md          # 阶段1：设计决策
├── todo.md            # 阶段2：任务计划
├── progress.md        # 阶段3：上下文快照（跨会话续跑用）
├── review.md          # 阶段5：评审记录
├── lessons.md         # 阶段6：教训（活跃）
└── lessons-archive.md # 阶段6：教训（归档）
```

## 核心原则

**简单优先** · **找根因不打补丁** · **只改需要改的** · **证明完成再宣布完成** · **流程是护栏**
