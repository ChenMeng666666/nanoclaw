# 2026-03-24 阶段 0：蓝图治理与边界统一详细规划

## 1. 阶段定位

阶段 0 不是功能开发阶段，而是整个蓝图的治理底座阶段。

它的任务不是“先做一点功能”，而是先回答下面这些根问题：

- 系统里谁是主体，谁只是上下文
- 什么数据属于谁，能流向哪里
- 什么能力应该进入核心，什么能力必须做成 skill
- 什么内容可以沉淀为记忆，什么内容可以升级为经验，什么内容可以进一步上行
- 后面每一个阶段到底以什么标准算完成

如果阶段 0 不先统一，后续阶段 1 到阶段 5 会出现主体归属混乱、数据边界模糊、能力放错层、知识上行失控的问题。

## 2. 阶段目标

本阶段的目标不是产出功能，而是产出一套后续迭代必须服从的治理框架。

### 2.1 目标一：统一术语

把 agent、group、memory、reflection、evolution、social、upward flow 等核心概念统一定义，避免多人或多轮迭代中同词异义。

### 2.2 目标二：统一边界

明确主体边界、数据边界、权限边界、扩展边界、上行边界，避免后续架构相互污染。

### 2.3 目标三：统一触发机制

明确记忆写入、经验提炼、反思归档、闲时触发、知识上行分别由什么事件触发。

### 2.4 目标四：统一验收标准

让后续每个阶段都能按统一标准判断“是否真的完成”，而不是只完成了局部实现。

## 3. 阶段产物

阶段 0 完成后，至少应形成以下治理产物：

1. **术语与对象模型文档**
2. **核心与扩展边界文档**
3. **记忆分层与生命周期文档**
4. **数据分类与权限矩阵文档**
5. **反思沉淀与经验升级文档**
6. **技能准入与上行规则文档**
7. **阶段验收标准文档**
8. **缺口清单与后续落位清单**

这意味着阶段 0 的结果更像“系统宪法”和“路线前置协议”，而不是代码模块。

## 4. 阶段 0 的核心治理问题

### 4.1 主体问题

要先明确：

- agent 是否是最小能力主体
- group 是运行上下文还是治理主体
- main 是全局控制者还是单独 agent
- 未来多 agent 社会里的“个体”“群体”“母体”分别是什么

如果这些不先定义，后续所有记忆、反思、任务记录、社交行为都无法准确归属。

### 4.2 记忆问题

要先明确：

- 什么是私有记忆
- 什么是共享记忆
- 什么是短期记忆
- 什么是长期记忆
- 什么是事实记忆
- 什么是关系记忆
- 什么是图谱层记忆

记忆系统不只是“存文本”，而是要提前定义未来的认知结构。

### 4.3 进化问题

要先明确：

- 什么算反思
- 什么算经验
- 什么算踩坑记录
- 什么算方法升级
- 什么内容只在个体内有效
- 什么内容可以变成群体经验

否则阶段 3 会退化成“生成很多总结”，但没有真正可复用的进化材料。

### 4.4 社会问题

要先明确：

- agent 之间的交互属于任务协作还是社会行为
- 闲时触发属于后台任务还是社会机制
- 社交工具是独立能力还是进化系统的外部接口

如果这些不先定义，阶段 4 很容易做成“多 agent 聊天”，但无法形成真正的局域智能体社会。

### 4.5 上行问题

要先明确：

- 哪些内容可以向上反哺主程序
- 哪些内容只能留在 agent 内部
- 哪些内容必须经过验证才能升级为规则
- 哪些内容永远不能直接上行

否则阶段 5 会把未成熟的局部经验误升级成系统机制。

## 5. 详细规划结构

建议把阶段 0 拆成 8 个子模块推进。

## 5.1 子模块 A：术语表与对象模型

### 目标

建立统一语言，作为所有后续文档和设计的基础。

### 必须定义的对象

- agent
- container
- group
- main
- memory item
- reflection item
- experience item
- social interaction
- upward knowledge
- skill
- core

### 必须回答的问题

- 哪些是运行对象
- 哪些是治理对象
- 哪些是存储对象
- 哪些是演进对象

### 输出要求

- 每个术语必须有唯一主定义
- 每个术语必须说明边界，不允许只写抽象愿景
- 每个术语必须说明与其他对象的关系

## 5.2 子模块 B：架构边界与核心/扩展划分

### 目标

明确什么属于 core，什么属于 custom，什么属于 skill，什么属于 group 内局部状态。

### 必须回答的问题

- 什么能力允许进入主程序
- 什么能力必须以 hook、event、middleware 方式接入
- 什么能力必须作为 skill 存在
- 什么能力只能存在于 agent 自己的数据域里

### 输出要求

- 一张核心 / 扩展 / 本地状态的边界表
- 一组禁入原则
- 一组 `[CUSTOM]` 标记使用原则

### 本模块的重要性

这个模块决定后续实现会不会污染上游核心，是整个蓝图的结构防火墙。

## 5.3 子模块 C：身份、归属与命名治理

### 目标

明确后续所有运行记录、记忆、反思、进化、社交行为的归属规则。

### 必须回答的问题

- 一个 agent 的唯一身份由什么组成
- agent 与 container 是一一对应还是运行时映射
- agent 与通道、模型、群组之间是什么关系
- group 中是否允许多个 agent 并存
- 一条记忆的属主是 agent、group 还是系统

### 输出要求

- 身份模型
- 归属模型
- 命名规则
- 冲突处理规则

### 本模块的重要性

阶段 1 的独立运行单元设计，必须直接建立在这个模块的结论之上。

## 5.4 子模块 D：记忆分层与生命周期治理

### 目标

明确记忆系统的分层模型和完整生命周期，作为阶段 2 的前置治理协议。

### 必须回答的问题

- 私有 / 共享 / 全局记忆如何分层
- 短期 / 长期记忆如何分层
- 事实 / 关系 / 图谱如何分层
- 写入、更新、检索、压缩、遗忘、归档各自如何触发
- 什么记忆可以被其他 agent 读取
- 什么记忆可以进入群体经验池

### 输出要求

- 一张分层模型图
- 一张生命周期流转表
- 一张读写权限矩阵

### 本模块的重要性

这是阶段 2 的制度前置，不先明确，后续很容易一边实现一边改概念。

## 5.5 子模块 E：数据分类、目录规范与权限矩阵

### 目标

把系统里的数据按所有权、敏感度、生命周期、是否可上行做分类。

### 必须回答的问题

- 哪些数据属于 agent 私域
- 哪些数据属于 group 共享域
- 哪些数据属于系统控制域
- 哪些数据属于宿主机敏感域
- 日志、会话、反思、进化数据集分别放在哪里
- 哪些数据允许挂载，哪些只允许代理访问

### 输出要求

- 数据分类表
- 目录落位原则
- 读写权限矩阵
- 上行许可矩阵

### 本模块的重要性

这个模块同时服务于安全、记忆、进化和社会协作，是阶段 0 的硬边界部分。

## 5.6 子模块 F：反思沉淀、经验升级与归档机制

### 目标

把“防重复踩坑”从一句原则变成可执行的治理流程。

### 必须回答的问题

- 哪些任务必须产出反思
- 反思的最小结构是什么
- 经验从反思中如何提取
- 哪些经验可以升级为规则
- 何时触发摘要与归档
- 如何让后续任务优先读取已有经验

### 输出要求

- 反思模板
- 经验提取规则
- 升级条件
- 归档阈值与摘要规则

### 本模块的重要性

这是阶段 3 的治理前置，也是蓝图里“共同进化”能否成立的决定点。

## 5.7 子模块 G：技能驱动准入与知识上行规则

### 目标

把“技能驱动”和“向上反哺”分别制度化，并建立两者之间的接口边界。

### 必须回答的问题

- 什么能力应该封装成 skill
- 什么能力不应该进入 skill
- skill 如何与记忆、反思、社交联动
- 什么知识允许上行
- 上行前需要什么验证
- 上行后会影响哪些系统层

### 输出要求

- skill 准入规则
- skill 测试要求
- 上行筛选标准
- 上行审批或确认机制

### 本模块的重要性

这个模块决定未来是“越来越模块化”，还是“越来越堆进核心”。

## 5.8 子模块 H：阶段验收标准与禁入事项

### 目标

为后续所有阶段建立统一验收方式，并提前写清楚哪些做法是不允许进入路线的。

### 必须回答的问题

- 阶段 1 完成的最低标准是什么
- 阶段 2 完成的最低标准是什么
- 阶段 3、4、5 各自的验收关键是什么
- 什么做法属于短期有效但长期破坏蓝图
- 什么情况必须暂缓推进

### 输出要求

- 各阶段验收清单
- 各阶段禁入项
- 依赖未满足时的暂缓规则

### 本模块的重要性

它负责防止后续开发“看起来做了很多，实际偏离蓝图”。

## 6. 阶段 0 的推进顺序

建议按下面的内部顺序推进，而不是并行发散：

1. **先定义术语与对象模型**
2. **再定义架构边界与身份归属**
3. **再定义记忆分层与数据边界**
4. **再定义反思、经验与归档机制**
5. **再定义 skill 准入与知识上行规则**
6. **最后定义各阶段验收标准与禁入项**

原因很简单：

- 没有术语，后面所有文档会混乱
- 没有边界，后面所有设计会互相侵入
- 没有数据与记忆模型，进化和社交无法建立在统一对象上
- 没有验收标准，后续阶段容易失控

## 7. 阶段 0 的当前缺口清单

结合当前仓库现状，阶段 0 至少暴露出以下治理缺口：

### 7.1 上位文档缺口

- 缺少统一的 `ROADMAP.md`
- 缺少可持续更新的 `TODO.md`
- 缺少 `ARCHITECTURE.md`
- 缺少 `docs/REFLECTIONS.md`

### 7.2 模型缺口

- 还没有正式的 agent 身份模型
- 还没有正式的记忆分层模型
- 还没有正式的经验升级模型
- 还没有正式的向上知识筛选模型

### 7.3 治理缺口

- 还没有统一的触发器定义
- 还没有统一的上行边界
- 还没有统一的验收标准
- 还没有统一的禁入项清单

## 8. 阶段 0 的完成标准

阶段 0 结束时，应满足以下标准：

- 所有核心术语都有唯一且稳定的定义
- 所有关键数据类型都有归属边界和流向边界
- 后续阶段 1 到 5 都有可引用的治理前提
- 对“什么能做、什么暂缓、什么不能做”有明确判断标准
- 后续写任何方案时，不再需要重新争论基础概念

如果这些条件还没满足，就说明阶段 0 还没有真正完成。

## 9. 阶段 0 之后的直接衔接

阶段 0 完成后，不应立刻全面铺开，而应按下面顺序衔接：

1. 先进入**阶段 1：Agent 独立运行单元**的详细规划
2. 只细化与阶段 1 直接相关的主体、配置、运行边界
3. 暂不提前实现阶段 3、4、5 的机制，只保留接口和治理占位

这样做的目的是保证：

- 先把主体立住
- 再把记忆立住
- 再把进化和社会能力挂上去

## 10. 下一轮建议拆解

下一轮建议继续只做规划，不写代码，优先拆成以下三个文档：

1. **阶段 0A：术语表与对象模型**
2. **阶段 0B：架构边界、身份归属与数据边界**
3. **阶段 0C：记忆生命周期、反思进化与知识上行规则**

如果要继续推进，最合理的顺序是先写 **阶段 0A**。
