# 2026-03-24 阶段 0A：术语表与对象模型

## 1. 文档目的

本文件用于给整个蓝图建立统一语言。

在后续阶段 1 到阶段 5 中，所有设计、规划、实现与验收，都默认以本文件中的术语和对象关系为准。

本文件优先解决三个问题：

1. **谁是主体**
2. **谁是上下文**
3. **谁拥有数据与能力**

如果这三个问题不先统一，后续记忆、进化、社交与知识上行都会出现归属混乱。

## 2. 建模原则

本阶段先采用“最少但稳定”的对象模型，不追求一次性把所有未来细节建满，而是先把后续一定会反复使用的核心对象固定下来。

### 2.1 原则一：主体优先

所有记忆、任务、反思、经验、社交行为都必须能追溯到明确主体。

### 2.2 原则二：上下文不等于主体

group、channel、container 都可能是运行上下文，但不天然等于能力主体。

### 2.3 原则三：数据必须有属主

任何一份数据都必须回答：

- 属于谁
- 谁可读
- 谁可写
- 是否可上行

### 2.4 原则四：先区分对象类型，再谈对象关系

先区分运行对象、治理对象、存储对象、演进对象，再定义它们之间的关系，避免术语交叉污染。

## 4. 核心术语表

以下术语为本蓝图的第一层基础定义。

## 4.1 agent

### 主定义

agent 是系统中的最小能力主体。

它是记忆归属、任务执行、反思产出、经验积累、社交行为与知识上行的第一责任单元。

### 边界

- agent 不是 container，但可运行在 container 中
- agent 不是 group，但可存在于某个 group 的上下文中
- agent 不是 channel，但可通过 channel 与外界通信
- agent 不等于模型配置，但必须绑定一套模型配置

### 关系

- agent 运行于一个 container 实例
- agent 绑定至少一套模型配置
- agent 可挂接一个或多个 channel 接入
- agent 可进入一个或多个 group 上下文
- agent 产生 memory、reflection、experience、social interaction

### 当前阶段的默认判断

后续蓝图默认采用：**agent 是最小主体，其他对象围绕 agent 建立归属关系。**

## 4.2 container

### 主定义

container 是 agent 的隔离运行载体。

### 边界

- container 是运行环境，不是能力主体
- container 可以销毁和重建，但 agent 身份不应因此消失
- container 负责隔离，不负责定义记忆归属

### 关系

- container 承载 agent 的一次或多次运行
- container 通过挂载访问有限数据域
- container 与 host 之间必须保持最小暴露面

### 当前阶段的默认判断

蓝图目标是“1 agent 1 容器”作为长期方向，但在术语层面要先区分：**主体是 agent，容器是运行壳。**

## 4.3 channel

### 主定义

channel 是 agent 与外部世界交互的输入输出通道。

### 边界

- channel 不是主体
- channel 不拥有核心记忆
- channel 只负责消息接入、身份映射与消息送达

### 关系

- channel 可绑定到一个或多个 agent
- channel 中的消息可被路由进入某个 group 上下文
- channel 承担外部身份到内部对象的映射责任

### 当前阶段的默认判断

channel 是通信层对象，不是认知层对象。

## 4.4 group

### 主定义

group 是 agent 运行时获得上下文、权限边界与共享资源视图的协作空间。

### 边界

- group 是上下文单元，不是最小能力主体
- group 可以承载共享记忆，但不能替代 agent 的私有身份
- group 可以是单人、自聊、群聊或未来的多 agent 协作空间

### 关系

- group 承载会话上下文
- group 挂接共享记忆与文件域
- group 约束 agent 可见信息和权限范围

### 当前阶段的默认判断

group 是协作和隔离单元，不是主体单元。

## 4.5 main

### 主定义

main 是系统最上层的建设、修复与治理入口。

### 边界

- main 不等于普通对话 group，也不应直接进入 agent 所在组参与对话治理
- main 不自动等同于单独 agent
- main 拥有建设、修复、审核和高权限配置能力
- main 可以处理全局记忆、注册组、调度任务、修复异常与高权限变更

### 关系

- main 是最上层治理入口，而不是组内协作角色
- main 与 global memory、修复流程、上行审核直接相关
- main 不负责日常 agent 对话治理，组内对话仍属于 agent 与 group 上下文
- main 是后续“向上知识反哺”的人工治理入口

### 当前阶段的默认判断

在当前蓝图中：**main = 最上层建设者与修复者入口，不是新的主体类别，也不是组内对话治理者。**

## 4.6 session

### 主定义

session 是某个上下文中的连续对话状态载体。

### 边界

- session 不是长期记忆本体
- session 保存对话连续性，但不等于稳定知识
- session 可被压缩、恢复、替换或归档

### 关系

- session 通常附着在 group 上下文中
- session 为 agent 提供短期连续性
- session 可成为反思与记忆提炼的输入材料

### 当前阶段的默认判断

session 属于短期运行状态对象，不属于长期治理对象。

## 4.7 task

### 主定义

task 是由 agent 或系统触发的、具备明确输入与执行边界的工作单元。

### 边界

- task 不是主体
- task 是行为实例，不是长期能力
- 一个 task 可以产生日志、反思、经验和新记忆

### 关系

- task 由 agent 执行
- task 发生于某个 group 或系统上下文
- task 是 evolution loop 的主要输入来源之一

### 当前阶段的默认判断

task 是演进材料采集的基本事件单位。

## 4.8 memory item

### 主定义

memory item 是被系统认定为值得在未来继续使用的信息单元。

### 边界

- memory item 不等于原始对话
- memory item 不等于反思
- memory item 可面向事实、偏好、关系、约束或任务背景

### 关系

- memory item 属于某个 agent、group 或更高层共享域
- memory item 可被检索、压缩、更新、遗忘和归档
- memory item 是 experience item 的原料之一，但不天然等于经验

### 当前阶段的默认判断

memory item 是“可复用信息单元”，不是“全部历史记录”。

## 4.9 reflection item

### 主定义

reflection item 是针对一次任务、失败、决策或执行过程形成的结构化复盘单元。

### 边界

- reflection item 必须带有原因分析，不只是总结
- reflection item 不等于普通记忆
- reflection item 不等于最终规则

### 关系

- reflection item 通常由 task 驱动产生
- reflection item 可提炼成 experience item
- reflection item 可进入归档或摘要系统

### 当前阶段的默认判断

reflection item 是进化链路的中间层对象。

## 4.10 experience item

### 主定义

experience item 是从 reflection、日志与历史成败中提炼出的可迁移方法单元。

### 边界

- experience item 必须具备复用价值
- experience item 不只是事实陈述
- experience item 尚未必然升级为系统规则

### 关系

- experience item 来源于 reflection、memory、task result
- experience item 可在个体、群体、系统之间逐级升级
- experience item 是向上知识反哺的重要候选对象

### 当前阶段的默认判断

experience item 是“可迁移方法”，不是“原始记录”。

## 4.11 knowledge item

### 主定义

knowledge item 是经过筛选、稳定化和语义收敛后，可长期复用的知识单元。

### 边界

- knowledge item 应比 memory 更稳定
- knowledge item 应比 experience 更少场景依赖
- knowledge item 可成为上行候选

### 关系

- knowledge item 可由稳定经验升级而来
- knowledge item 可服务于全局策略、规范和系统设计

### 当前阶段的默认判断

knowledge item 是“稳定知识层”，位于经验之上、规则之下。

## 4.12 graph relation

### 主定义

graph relation 是两个或多个对象之间被结构化表达的关联关系。

### 边界

- graph relation 不是单独记忆文本
- graph relation 不是完整图谱系统本身
- 它是未来关系型记忆与图谱层的基本连接件

### 关系

- graph relation 可连接 agent、memory、experience、task、concept
- graph relation 支撑后续记忆系统的关系理解能力

### 当前阶段的默认判断

图谱不是第一层对象，**关系才是图谱的最小结构单元。**

## 4.13 skill

### 主定义

skill 是封装独立能力、外部交互或后台流程的可调用扩展单元。

### 边界

- skill 不是主体
- skill 不应替代核心治理规则
- skill 不应直接篡改主循环核心

### 关系

- skill 被 agent 调用
- skill 承担扩展能力
- skill 可读取受权限约束的数据
- skill 可参与记忆、反思、社交或任务流程

### 当前阶段的默认判断

复杂后台任务、外部交互、反思机制、社会机制优先做成 skill，而不是写死进 core。

## 4.14 core

### 主定义

core 是系统最小稳定主干，负责运行编排、基础安全边界、核心调度与必要的全局机制。

### 边界

- core 不承载高频变化的实验能力
- core 不应直接硬编码复杂定制逻辑
- core 必须保持尽量稳定与可同步上游

### 关系

- core 通过事件、钩子、中间件或标准接口与 custom / skill 协作
- core 是所有扩展能力的基础承载层

### 当前阶段的默认判断

core 只保留必须稳定存在的系统骨架，变化快的能力尽量外置。

## 4.15 custom module

### 主定义

custom module 是为本项目蓝图引入、但不属于上游原生核心的定制化模块。

### 边界

- custom module 不等于 skill
- custom module 通常比 skill 更接近系统结构层
- custom module 必须与上游核心保持隔离

### 关系

- custom module 可为 skill、memory、evolution、social 提供结构支撑
- custom module 受 `[CUSTOM]` 规则与隔离原则约束

### 当前阶段的默认判断

当能力需要长期存在但又不适合进入 core 时，优先考虑 custom module。

## 4.16 social interaction

### 主定义

social interaction 是 agent 与其他 agent、群体或治理节点之间发生的结构化交互行为。

### 边界

- social interaction 不等于普通消息传递
- social interaction 必须带有对象关系、意图或协作语义
- social interaction 不天然等于任务协作，但可包含任务协作

### 关系

- social interaction 可成为记忆和经验输入
- social interaction 是 Private Moltbook 的最小行为单元

### 当前阶段的默认判断

社会化不是“能发消息”，而是“交互可以沉淀为关系和协作结构”。

## 4.17 evolution loop

### 主定义

evolution loop 是系统从执行、复盘、提炼到方法升级的闭环过程。

### 边界

- evolution loop 不是单个对象，而是过程对象
- evolution loop 不依赖一次性 prompt，总体上应可持续反复运行

### 关系

- 输入来自 task、session、memory、reflection、social interaction
- 输出指向 experience、knowledge、archive summary、upward knowledge

### 当前阶段的默认判断

进化不是额外功能，而是贯穿任务生命周期的闭环流程。

## 4.18 upward knowledge

### 主定义

upward knowledge 是从个体或群体层沉淀后，被筛选为可反哺更高系统层的知识产物。

### 边界

- upward knowledge 不等于任意经验
- upward knowledge 必须经过筛选和验证
- upward knowledge 不应直接包含高噪声、强场景依赖内容

### 关系

- upward knowledge 来源于稳定 knowledge item 或 experience item
- upward knowledge 可影响 core policy、skill policy、架构规则与工作流

### 当前阶段的默认判断

向上反哺的前提不是“新”，而是“稳定、可迁移、值得进入母体”。

## 4.19 archive summary

### 主定义

archive summary 是对长日志、长记忆、长反思进行高密度压缩后形成的摘要对象。

### 边界

- archive summary 不是原始数据替身
- archive summary 目标是降低 token 消耗和检索成本
- archive summary 必须保留核心原则与关键关系

### 关系

- archive summary 来自 memory、reflection、log、dataset 的归档过程
- archive summary 是防遗忘与控上下文成本的重要中间层

### 当前阶段的默认判断

主程序默认应优先消费摘要层，而不是无上限消费原始历史。

## 4.20 policy

### 主定义

policy 是系统在治理、权限、边界、准入、上行和验收方面的规则对象。

### 边界

- policy 不等于实现代码
- policy 不等于一次性建议
- policy 需要能被后续文档、实现和验收复用

### 关系

- policy 约束 core、custom module、skill、agent、data flow
- policy 是阶段 0 的核心产物之一

### 当前阶段的默认判断

阶段 0 最终产出的本质，就是一套可复用 policy 集合。

## 5. 对象关系模型

为了避免术语分散，后续统一采用下面这组主关系。

### 5.1 主体关系

- agent 是最小主体
- group 是协作上下文
- main 是特权 group
- container 是运行载体
- channel 是通信接口

### 5.2 行为关系

- agent 执行 task
- task 产生日志、反思、记忆变更与经验候选
- agent 发起或参与 social interaction
- evolution loop 消费历史材料并产出升级结果

### 5.3 数据关系

- memory item 保存可复用信息
- reflection item 保存结构化复盘
- experience item 保存可迁移方法
- knowledge item 保存稳定知识
- graph relation 表达这些对象之间的连接
- archive summary 控制成本并维持可检索性

### 5.4 治理关系

- core 承担稳定骨架
- custom module 承担结构化定制
- skill 承担独立扩展能力
- policy 负责约束所有边界
- upward knowledge 负责把稳定成果上送更高层

## 6. 关键反混淆规则

为了防止后续规划再次混淆，先固定以下规则：

### 6.1 主体与上下文分离

- agent 是主体
- group 不是主体
- container 不是主体
- channel 不是主体

### 6.2 记忆与反思分离

- 记忆强调未来可调用的信息
- 反思强调对执行过程的复盘和归因
- 经验强调跨任务复用的方法

### 6.3 经验与知识分离

- 经验更贴近做法
- 知识更贴近稳定认知
- 上行知识必须来自更稳定的一层，而不是直接来自原始日志

### 6.4 core 与 skill 分离

- core 保持稳定
- skill 承担变化和扩展
- custom module 承担长期存在但不宜进 core 的结构能力

## 7. 当前默认对象模型

如果后续文档没有特别声明，统一采用以下默认模型：

1. **agent 是最小能力主体**
2. **group 是协作上下文，不是主体**
3. **main 是特权 group，不是新主体**
4. **container 是 agent 的隔离运行壳**
5. **memory、reflection、experience、knowledge 是四层不同对象**
6. **graph relation 是图谱能力的最小结构单元**
7. **skill 是扩展能力单元，core 是稳定骨架**
8. **upward knowledge 只能来自经过筛选的稳定产物**

## 8. 本文对后续阶段的直接约束

### 对阶段 1 的约束

- 必须围绕 agent 身份建模
- 不能把 group 误当成主体
- 不能把 container 身份误当作 agent 身份

### 对阶段 2 的约束

- 记忆设计必须至少区分 memory、reflection、experience、knowledge
- 图谱设计必须以 relation 为最小结构，而不是先上巨大图谱系统

### 对阶段 3 的约束

- 进化流程必须建立在 task → reflection → experience 的链路上
- 不能把普通总结直接当作经验升级

### 对阶段 4 的约束

- 社交能力必须围绕 social interaction 的结构语义展开
- 不能把“能互发消息”误判为“已形成智能体社会”

### 对阶段 5 的约束

- 向上反哺必须经过筛选
- 不能直接把原始记忆或未经验证经验上送主程序
