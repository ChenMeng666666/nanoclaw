# 2026-03-24 阶段 0B：架构边界、身份归属与数据边界

## 1. 文档目的

本文件承接阶段 0A 的术语表，进一步回答三个问题：

1. **什么能力应该放在哪一层**
2. **什么对象拥有哪类权限与归属**
3. **什么数据应该落在哪个边界里**

阶段 0A 解决的是“名词定义”，阶段 0B 解决的是“边界与归位”。

如果阶段 0B 不先明确，后续阶段 1 与阶段 2 会同时出现三种风险：

- 把本该进 skill 的能力塞进 core
- 把本该属于 agent 的数据混进 group 或系统层
- 把运行态状态与源码、治理信息、长期知识混放

## 2. 本文采用的前置共识

本文件默认继承并固定以下已确认共识：

1. **agent 是最小能力主体**
2. **group 是协作上下文，不是主体**
3. **main 是最上层建设、修复与审核入口，不是新的主体类型**
4. **一个 group 未来允许承载多个 agent**
5. **upward knowledge 前期必须通过 main 作为人工治理入口**

这五点在后续边界矩阵中都视为已成立前提。

## 3. 边界设计总原则

## 3.1 稳定骨架与变化能力分离

- 稳定、基础、全局性的能力进入 core
- 长期存在但不适合进 core 的结构能力进入 custom module
- 易变化、可插拔、外部交互型能力进入 skill

## 3.2 主体与载体分离

- agent 是主体
- container 是载体
- group 是上下文
- channel 是接口

主体负责归属，载体负责运行，上下文负责隔离，接口负责通信。

## 3.3 权限与可见面绑定

一个对象能做什么，不由提示词决定，而由它的可见目录、宿主校验点和写权限决定。

## 3.4 现状与目标分离

本文件所有边界都同时区分：

- **目标规范**：蓝图希望长期稳定达到的边界
- **当前现状**：仓库已经落地的目录、挂载和权限收口方式

这样可以避免把“目标架构”误写成“当前实现事实”。

## 4. 第一层边界：架构分层边界

本节定义 core、custom module、skill、group local state 四层的边界。

## 4.1 分层主定义

### core

系统稳定骨架，负责运行编排、安全边界、核心调度、基础路由、最小全局机制。

### custom module

为本项目蓝图长期服务、但不适合进入上游 core 的结构化定制层。

### skill

被 agent 按需调用的能力扩展层，负责外部交互、后台流程、专项任务和可插拔功能。

### group local state

只在某个 group 或某个 agent 局部上下文中有效的本地状态层，不应冒充系统级结构。

## 4.2 架构边界矩阵

| 层级 | 主职责 | 是否稳定长期存在 | 是否允许高频变化 | 是否直接拥有治理权 | 典型落位 | 是否允许直接改主循环 |
|---|---|---|---|---|---|---|
| core | 编排、安全、调度、基础路由 | 是 | 否 | 是 | `src/` 核心稳定文件 | 否，除必要修补外应最小化 |
| custom module | 长期结构化定制能力 | 是 | 低到中 | 受 policy 约束 | 目标应独立于 core 落位 | 不应直接改，优先接口接入 |
| skill | 外部交互、后台流程、专项能力 | 否，按需存在 | 是 | 否 | `.claude/skills/` | 否 |
| group local state | group 或 agent 的局部状态 | 否 | 是 | 否 | `groups/`、局部数据目录 | 否 |

## 4.3 分层准入规则

### 进入 core 的条件

只有同时满足以下条件，才允许进入 core：

- 属于所有 agent 或所有运行流程都必须依赖的骨架能力
- 与安全边界、基础调度、核心路由或宿主校验直接相关
- 不适合作为 skill 按需加载
- 如果缺失，会导致整个系统无法稳定运行

### 进入 custom module 的条件

只有同时满足以下条件，才适合进入 custom module：

- 能力具有长期存在价值
- 能力不适合变成一次性 skill
- 能力又不应侵入上游 core
- 能为 memory、evolution、social、upward flow 提供结构层支撑

### 进入 skill 的条件

以下能力优先进入 skill：

- 外部通道接入
- 复杂后台任务
- 反思、归档、社会行为等专项机制
- 可替换、可独立发布、可按需启用的能力

### 停留在 group local state 的条件

以下内容应停留在局部状态层：

- 某个 group 的短期上下文材料
- 某个 agent 的局部实验配置
- 仅对个别协作空间有效的共享文件
- 不具备全局复用价值的临时产物

## 4.4 禁入规则

以下做法视为 0B 明确禁入：

- 把记忆处理、进化流程、社会化机制直接硬编码进主循环
- 让 skill 直接承担治理规则定义权
- 让 group local state 伪装成系统公共结构
- 用 container 运行边界替代 agent 身份边界
- 因实现方便而把运行态状态写回源码目录

## 5. 第二层边界：身份归属边界

本节回答“谁拥有谁”“谁能写什么”“谁能跨边界操作”。

## 5.1 身份归属总则

### 总则一：主体归属以 agent 为第一锚点

所有 task、memory、reflection、experience、social interaction，默认都先回答“属于哪个 agent”。

### 总则二：上下文归属以 group 为第二锚点

同一 agent 在不同 group 中运行，允许拥有不同上下文、不同共享资源视图和不同权限约束。

### 总则三：治理归属以 main 为高权限入口

main 不作为新主体，但承担建设、修复、审核、全局配置与知识上行入口。

## 5.2 身份与归属矩阵

| 对象 | 是主体吗 | 主要归属角色 | 可管理范围 | 可写范围 | 跨组能力 | 备注 |
|---|---|---|---|---|---|---|
| agent | 是 | 最小能力主体 | 自身任务、记忆、行为 | 受上下文约束的局部写入 | 由 policy 决定，不天然拥有 | 所有认知归属的第一锚点 |
| group | 否 | 协作上下文 | 本 group 共享视图 | 本 group 局部共享域 | 否 | 不是主体，只是隔离与协作空间 |
| main | 否 | 最上层建设与修复入口 | 修复、审核、全局配置、异常处理 | 全局记忆、高权配置、治理入口 | 受控地是 | 不直接进入组内参与日常对话治理 |
| container | 否 | 运行载体 | 运行期可见目录 | 挂载授权范围 | 否 | 不拥有认知归属 |
| channel | 否 | 外部通信接口 | 消息接入与发送 | 不应直接承担核心知识写入 | 条件性 | 负责身份映射，不负责治理 |
| skill | 否 | 扩展能力单元 | 自身执行范围 | 被授予的最小权限范围 | 不天然拥有 | 调用者仍是 agent |

## 5.3 关键归属规则

### 规则一：记忆先归 agent，再判断是否进入共享域

默认先问“这是哪个 agent 的记忆”，再决定它是否升级为 group 共享、全局共享或上行知识。

### 规则二：group 只承载共享，不吞并私有归属

即使某条信息在 group 中可见，也不代表它天然失去 agent 属主。

### 规则三：main 的高权来自治理地位，不来自主体升级

main 拥有更大的可见面和更强权限，但它的职责是建设、修复和审核，而不是直接下沉到组内充当对话治理者。

### 规则四：container 永远不能成为归属主键

container 可重建、迁移、销毁，因此不能作为记忆、反思、经验和知识的最终归属主键。

## 5.4 行为归属矩阵

| 行为 | 第一归属 | 第二归属 | 可否直接跨组 | 是否允许进入全局层 | 是否允许上行 |
|---|---|---|---|---|---|
| task 执行 | agent | group | 默认否 | 否 | 条件性 |
| memory 写入 | agent | group / global | 默认否 | 仅特定规则允许 | 条件性 |
| reflection 产出 | agent | task / group | 否 | 默认否 | 条件性 |
| experience 提炼 | agent | group / evolution loop | 条件性 | 条件性 | 是，需筛选 |
| social interaction | agent | group / social layer | 条件性 | 否 | 条件性 |
| upward knowledge 提交 | agent / group | main 治理入口 | 是，但必须经治理入口 | 是 | 是 |

## 6. 第三层边界：权限边界

本节把“谁能看到什么、谁能写什么、谁能跨边界操作”显式写出来。

## 6.1 权限主原则

- 权限由宿主侧边界和目录可见面决定
- 权限不是提示词承诺，而是挂载和校验结果
- 普通 group 默认按不可信上下文处理
- main 是唯一默认拥有高权限修复与审核入口的上下文

## 6.2 上下文权限矩阵

| 上下文 | 可见范围 | 可写范围 | 是否可跨组发消息 | 是否可跨组调度任务 | 是否可写全局记忆 | 是否可配置额外挂载 |
|---|---|---|---|---|---|---|
| main | 项目根只读视图 + 自身治理写域 + 全局治理可见域 | 被允许的修复、审核与治理写域 | 受控地是 | 是 | 是 | 是 |
| non-main group | 自身 group 可写域 + global 只读视图 | 自身 group 局部可写域 | 否 | 否 | 否 | 默认否 |
| skill 执行期 | 继承调用方上下文 | 继承最小必要权限 | 不天然拥有 | 不天然拥有 | 不天然拥有 | 否 |
| container runtime | 按挂载暴露目录 | 按挂载权限 | 否 | 否 | 否 | 否 |

## 6.3 跨边界操作规则

### main 允许的跨边界操作

- 为任意 group 调度修复性或治理性任务
- 管理注册 group
- 管理全局记忆
- 发起跨 group 的建设、修复与审核动作
- 作为上行知识的人工审核入口

### main 不应承担的职责

- 不直接进入某个 agent 所在组参与日常对话治理
- 不替代组内 agent 做常规协作决策
- 不把“拥有高权限”误用成“直接接管组内会话”

### non-main 禁止的跨边界操作

- 不得直接为其他 group 发送消息
- 不得直接为其他 group 调度任务
- 不得写入全局记忆
- 不得直接触发主程序规则变更

### skill 的受限原则

- skill 不因自身存在而升级调用方权限
- skill 只能在调用方既有权限内工作
- 若 skill 需要更高权限，必须通过明确治理入口而非隐式提权

## 7. 第四层边界：数据边界

本节定义源码域、治理域、组本地域、运行态域之间的边界。

## 7.1 数据域分类

### A. 源码域

保存系统实现代码、稳定文档、配置样例与可被版本控制的工程资产。

### B. 治理域

保存蓝图、规则、阶段规划、架构原则、反思制度和高层治理产物。

### C. 组本地域

保存 group 级共享记忆、局部文件、协作材料和与上下文绑定的产物。

### D. 运行态状态域

保存 session、IPC、数据库、认证态、日志、临时 env、未来进化数据集等运行过程中持续变化的数据。

## 7.2 数据域矩阵

| 数据域 | 典型内容 | 当前现状落点 | 目标规范落点 | 是否进版本控制 | 默认写入者 | 备注 |
|---|---|---|---|---|---|---|
| 源码域 | `src/`、`container/`、稳定配置 | 已存在 | 保持不变 | 是 | 开发者与治理流程 | 不应承载运行态状态 |
| 治理域 | `plan/`、顶层治理文档、未来 `ARCHITECTURE.md` | 已部分存在 | 明确收敛为治理入口 | 是 | main / 人工治理流程 | 用于规划、规则、修复决策与审核沉淀 |
| 组本地域 | `groups/{name}/` 的记忆与文件 | 已存在 | 保持为上下文共享域 | 仅少量模板可控 | agent / group | 不应吞并全局运行态 |
| 运行态状态域 | `data/`、`store/`、`logs/` | 已存在但分散 | 长期收敛到数据域规范下 | 否 | 系统运行时 | 必须与源码域强隔离 |

## 7.3 数据对象归位矩阵

| 数据对象 | 第一落位 | 第二落位 | 可见范围 | 是否允许上行 | 当前判断 |
|---|---|---|---|---|---|
| global memory | 治理域 / 全局记忆域 | 受控共享域 | main 可写，全局可读 | 条件性 | 属于高权共享层 |
| group memory | 组本地域 | 共享上下文域 | 本 group 可见 | 条件性 | 属于协作共享层 |
| agent 私有记忆 | 未来应独立于 group 表达 | 可投影到组上下文 | 默认仅属主 agent 可完全控制 | 条件性 | 当前实现尚未完全独立 |
| reflection | 运行态与治理中间层 | 可归档到反思库 | 默认受限 | 是 | 先作为中间产物 |
| experience | 经验域 | 群体经验池 | 条件性共享 | 是 | 可升级但需筛选 |
| upward knowledge | 治理域 | 主程序规则入口 | main 审核后可见 | 是 | 不允许直接自动上送 |
| session transcript | 运行态状态域 | 归档摘要层 | 默认受限 | 否 | 属于短期连续性材料 |
| task log | 运行态状态域 | 反思输入层 | 默认受限 | 否 | 不等于稳定知识 |
| graph relation | 未来结构化记忆域 | 可被摘要与检索消费 | 条件性 | 条件性 | 是图谱层基础件 |

## 7.4 数据边界禁令

- 不允许把数据库、认证态、日志写进源码域
- 不允许把原始会话直接当作长期知识层
- 不允许让 group 文件夹承担全部未来运行态数据
- 不允许让上行知识直接来自未经筛选的原始日志
- 不允许把未来图谱数据与临时文本杂糅成同一层

## 8. 第五层边界：挂载与宿主安全边界

本节明确“边界由什么落实”。

## 8.1 落实手段

边界最终不是靠文档，而是靠以下四类机制落地：

1. **容器挂载范围**
2. **宿主侧 IPC 权限校验**
3. **allowlist 与只读约束**
4. **数据目录与版本控制隔离**

## 8.2 挂载规则

### main 的目标规则

- 可拥有更大只读视野
- 可拥有自身治理写域
- 可通过受控入口管理全局资源

### non-main 的目标规则

- 只看见自身上下文必要目录
- 对全局层默认只读
- 对额外挂载默认受限且保守

### 额外挂载规则

- 必须显式 allowlist
- 必须可审计
- 非 main 默认只读
- 任何宿主敏感信息禁止直接暴露

## 8.3 安全边界结论

蓝图后续所有设计都必须接受一个硬前提：

**真正的安全边界不是“agent 说自己不会做”，而是“宿主根本没把不该给它的东西挂进去”。**

## 9. 目标规范与当前现状的差异

为了避免后续误读，这里明确列出差异。

## 9.1 已经基本落地的边界

- main 与 non-main 的权限差异已经存在，但当前实现里的 main 更接近管理通道，蓝图中需进一步收敛为建设与修复入口
- container 可见面和挂载隔离已经存在
- group 目录与全局记忆分层已经存在
- 运行态数据与源码大体分离已经存在

## 9.2 尚未完全落地的目标边界

- custom module 还没有稳定目录与正式落位
- agent 私有记忆还没有完全脱离 group 表达
- experience、knowledge、upward knowledge 还没有正式持久化分层
- 进化数据集与图谱数据还没有正式目录规范
- 反思库与架构日记仍缺正式入口文档

## 9.3 当前阶段的处理原则

对于“目标已清楚、实现未完成”的部分，后续规划统一采用：

- **先按目标规范写边界**
- **同时在文档中标出现状缺口**
- **后续实现必须朝目标规范收敛，而不是倒过来让现状定义蓝图**

## 10. 阶段 0B 的结论清单

本阶段先固定以下结论，后续文档默认直接继承：

1. **core 只承载稳定骨架，不承载高频变化能力**
2. **custom module 是长期结构定制层，但必须与上游核心隔离**
3. **skill 是扩展能力层，不拥有治理权，也不天然提升权限**
4. **group local state 只能代表局部上下文，不能冒充系统层**
5. **agent 是认知归属主键，container 不能成为归属主键**
6. **main 是唯一默认高权限建设、修复与审核入口**
7. **non-main 默认按不可信上下文处理**
8. **数据必须分为源码域、治理域、组本地域、运行态状态域**
9. **上行知识必须先进入治理入口，不允许原始自动上送**
10. **后续实现一律以宿主收口边界为准，不以 prompt 承诺代替权限**

## 11. 本文对后续阶段的直接约束

### 对阶段 1 的约束

- agent 身份设计必须独立于 container 生命周期
- agent 配置设计必须与 group 上下文解耦
- 多 agent 同 group 必须预留边界

### 对阶段 2 的约束

- 记忆层必须区分 agent 私域、group 共享域、全局共享域
- 记忆与反思、经验、知识不得继续混层

### 对阶段 3 的约束

- 反思与经验必须进入独立中间层，不能直接挤进 group memory
- 进化数据必须进入运行态或专门数据域，而不是源码域

### 对阶段 4 的约束

- social interaction 必须建立在“多主体 + 共享上下文 + 受控权限”三者同时成立的前提上
- 社会层不得绕开 main 的治理入口直接改写全局层

### 对阶段 5 的约束

- upward knowledge 必须经过 main 治理入口筛选
- 主程序规则不得直接消费原始日志和原始会话

## 12. 下一步建议

下一轮建议继续只做规划，不写代码，直接推进：

1. **阶段 0C：记忆生命周期、反思进化与知识上行规则**
