# 2026-03-24 阶段 0C：记忆生命周期、反思进化与知识上行规则

## 1. 文档目的

本文件承接阶段 0A 与阶段 0B，目标是把下面这条链路正式制度化：

**task / session / log → memory → reflection → experience → knowledge → upward knowledge**

同时，本文件还要回答另外两个核心问题：

1. **每一层对象什么时候产生**
2. **每一层对象在什么条件下才能升级、归档、被读取或上行**

阶段 0A 解决“对象叫什么”，阶段 0B 解决“对象放在哪、归谁管”，阶段 0C 解决“对象怎么流动”。

## 2. 本文采用的前置共识

本文件默认继承以下已确认前提：

1. **agent 是最小能力主体**
2. **group 是协作上下文，不是主体**
3. **main 是最上层建设、修复与审核入口，不参与组内日常对话治理**
4. **container 是运行壳，不是归属主键**
5. **memory、reflection、experience、knowledge 是不同层级对象**
6. **upward knowledge 必须通过 main 所代表的治理平面进行审核**

## 3. 生命周期设计原则

## 3.1 原始材料不能直接上行

task、session、log 都只是原始材料，不能直接等同于经验、知识或系统规则。

## 3.2 每一层只解决一个问题

- memory 解决“以后还要不要记住”
- reflection 解决“为什么会这样”
- experience 解决“以后怎么做更好”
- knowledge 解决“什么已经稳定成立”
- upward knowledge 解决“什么值得反哺母体”

## 3.3 升级必须比写入更严格

写入记忆的门槛可以低于经验升级，经验升级的门槛必须低于知识上行。

越往高层走，噪声越低，验证越严格，数量越少。

## 3.4 读取优先级必须高于重复生成

在生成新反思、新经验、新规则前，系统应优先读取已有摘要、已有反思、已有知识，避免重复踩坑和重复消耗 token。

## 3.5 主程序默认消费摘要层，而不是原始历史层

长日志、长记忆、长反思必须经归档与摘要处理后，才成为主程序日常读取对象。

## 4. 生命周期总览

## 4.1 七层对象流转

本阶段统一采用以下七层流转：

1. **原始材料层**
2. **记忆层**
3. **反思层**
4. **经验层**
5. **知识层**
6. **摘要归档层**
7. **上行治理层**

## 4.2 总体流转图

```text
task / session / log / social interaction
                ↓
           memory item
                ↓
         reflection item
                ↓
         experience item
                ↓
          knowledge item
                ↓
         upward knowledge

长日志 / 长记忆 / 长反思 / 长经验
                ↓
         archive summary
                ↓
     成为后续读取的主入口之一
```

## 5. 各层定义与职责

## 5.1 原始材料层

### 包含对象

- task result
- session transcript
- runtime log
- social interaction record
- 用户显式输入

### 职责

- 保留事实现场
- 提供后续记忆与反思的输入
- 作为失败分析和经验提炼的原始依据

### 非职责

- 不直接充当长期知识
- 不直接上行
- 不直接当作系统规则

## 5.2 记忆层

### 包含对象

- fact memory
- preference memory
- constraint memory
- relation memory
- context memory

### 职责

- 保存未来还需要再次用到的信息
- 降低 agent 对上下文的瞬时依赖
- 为反思和经验提供可检索材料

### 非职责

- 不负责解释原因
- 不负责总结方法
- 不直接等于经验与知识

## 5.3 反思层

### 包含对象

- bug reflection
- task reflection
- failure reflection
- decision reflection

### 职责

- 对结果进行归因
- 对错误进行结构化复盘
- 对方案优劣进行可复用分析

### 非职责

- 不替代记忆
- 不等于最终经验
- 不直接上行为系统规则

## 5.4 经验层

### 包含对象

- 坑点规避方法
- 成功模式
- 失败模式
- 可迁移步骤
- 决策建议

### 职责

- 提炼可跨任务复用的方法
- 形成 agent 或群体层的能力增益
- 为知识层提供候选素材

### 非职责

- 不要求完全稳定
- 不应包含大量原始叙事
- 不应直接取代知识层

## 5.5 知识层

### 包含对象

- 稳定规则
- 稳定认知
- 长期适用原则
- 结构性判断

### 职责

- 沉淀更稳定、更抽象、更少场景依赖的结论
- 为治理平面和后续系统设计提供可复用支撑

### 非职责

- 不保存原始日志
- 不保存一次性经验
- 不自动进入主程序规则

## 5.6 摘要归档层

### 包含对象

- archive summary
- 核心原则摘要
- 高频坑点摘要
- 长会话压缩结果

### 职责

- 降低 token 消耗
- 保留高密度信息
- 为“行动前优先读取”提供轻量入口

### 非职责

- 不作为原始记录的唯一替代
- 不制造新知识，只压缩现有材料

## 5.7 上行治理层

### 包含对象

- upward knowledge candidate
- main 审核结果
- 已批准的反哺产物

### 职责

- 筛选值得反哺母体的稳定成果
- 把局部经验升级为治理级知识候选
- 控制知识进入主程序的质量

### 非职责

- 不直接参与组内对话治理
- 不接收未经筛选的原始材料
- 不以“高权限”替代验证过程

## 6. 生命周期状态机

## 6.1 memory item 状态

| 状态 | 含义 | 进入条件 | 离开条件 |
|---|---|---|---|
| captured | 已捕获但未整理 | 发现可复用信息 | 分类完成或丢弃 |
| normalized | 已结构化 | 完成归属与类型整理 | 被读取、更新或归档 |
| shared | 已进入共享视图 | 满足共享条件 | 被降级、更新或归档 |
| archived | 已进入摘要/归档 | 超长、低频或被替换 | 被重新激活或长期保留 |
| deprecated | 已过时 | 被新信息替代 | 删除或保留历史痕迹 |

## 6.2 reflection item 状态

| 状态 | 含义 | 进入条件 | 离开条件 |
|---|---|---|---|
| required | 必须产出反思 | 触发强制反思事件 | 写完反思 |
| drafted | 已形成初稿 | 已完成初步复盘 | 提炼经验或归档 |
| extracted | 已完成经验提炼 | 形成可迁移方法 | 进入摘要或保留 |
| archived | 已归档 | 超长或历史完成 | 被检索引用 |

## 6.3 experience item 状态

| 状态 | 含义 | 进入条件 | 离开条件 |
|---|---|---|---|
| candidate | 经验候选 | 反思中发现可复用方法 | 验证或淘汰 |
| validated | 已验证 | 经重复任务、对照结果或人工确认 | 升级为知识或归档 |
| adopted | 已纳入常用方法 | 被后续任务成功复用 | 继续稳定或升级 |
| archived | 已归档 | 被新经验替换或低频使用 | 被再激活 |

## 6.4 knowledge item 状态

| 状态 | 含义 | 进入条件 | 离开条件 |
|---|---|---|---|
| candidate | 知识候选 | 多条经验收敛到稳定结论 | 审核或淘汰 |
| stabilized | 已稳定 | 具有长期适用性 | 进入上行候选或归档 |
| upward_candidate | 上行候选 | 满足上行条件 | 审核批准或驳回 |
| archived | 已归档 | 被更高质量知识替代 | 被再引用 |

## 7. 触发器体系

## 7.1 写入触发器

以下情形触发 memory 写入：

- 用户显式说“记住这个”
- 出现未来高概率复用的事实、偏好、约束
- 某个关系信息对后续任务持续有价值
- 某个 group 形成稳定共享上下文
- 某个 agent 形成应长期保存的私有背景

## 7.2 反思触发器

以下情形强制或优先触发 reflection：

- 解决复杂 Bug
- 失败后完成修复
- 完成阶段性任务
- 出现明显错误路径或踩坑记录
- 做出关键架构决策
- 同类问题出现第二次时

## 7.3 经验提炼触发器

以下情形触发 experience 提炼：

- 反思中出现可迁移方法
- 有可比较的成功与失败样本
- 某方法在不同任务中重复奏效
- 某坑点具备高复发风险

## 7.4 知识升级触发器

以下情形允许 experience 升级为 knowledge：

- 同类经验跨多个任务仍成立
- 结论已明显弱化场景依赖
- 可抽象成稳定原则或规则
- 经治理平面或人工审核认可

## 7.5 归档触发器

以下情形必须触发 archive summary：

- 日志、记忆、反思文档超过 500 行
- 同类材料持续堆积导致检索成本过高
- 某对象已低频使用但仍需保留
- 某对象已被更新版本替代，但历史脉络仍有价值

## 7.6 读取触发器

以下情形必须优先读取摘要、反思或知识层：

- 开始新的编码任务前
- 遇到类似 bug 或相似任务前
- 准备做重大架构决策前
- 准备写入新的高层知识前
- 准备上行前

## 7.7 上行触发器

只有同时满足以下条件，才允许生成 upward knowledge candidate：

- 已不是原始日志或原始会话
- 已经过经验或知识层稳定化
- 具备可迁移价值
- 对主程序或治理规则具有增益
- 已经过 main 所代表治理平面的审核流程

## 8. 读写与升级规则

## 8.1 写入规则

- memory 的写入门槛最低，但必须有明确属主
- reflection 不因“想总结一下”自动产生，必须与任务或问题绑定
- experience 不得从纯臆测生成，必须来自反思或可对照材料
- knowledge 不得从单次任务结论直接升级
- upward knowledge 不得从未经验证经验直接上送

## 8.2 共享规则

- agent 私有记忆默认不自动进入 group 共享层
- group 共享记忆默认不自动进入全局层
- 全局层默认不自动进入主程序规则层
- 每次升级共享范围，都必须经过更严格筛选

## 8.3 替换规则

- 新版本对象应优先替换旧版本对象的主读取入口
- 旧版本对象不一定删除，可进入 archived
- 替换必须保留最小历史可追踪性

## 8.4 删除规则

只有在以下情形才允许彻底删除：

- 明确错误且无保留价值
- 已被上位稳定对象完全覆盖
- 涉及敏感内容且不应继续保留

## 9. 归属与可见性规则

## 9.1 memory 的归属

- 第一归属：agent
- 第二归属：group 或共享层
- 例外：全局记忆只能通过受控治理入口处理

## 9.2 reflection 的归属

- 第一归属：触发该任务的 agent
- 第二归属：任务或治理记录
- 不直接属于 group 会话本身

## 9.3 experience 的归属

- 初期归属：agent
- 成熟后可进入 group 或群体经验池
- 进一步验证后才可能进入知识层

## 9.4 upward knowledge 的归属

- 候选来源可以来自 agent 或 group
- 审核与批准只属于 main 所代表的治理平面
- 已批准产物属于更高层治理资产，而不是组内会话资产

## 10. 摘要与归档政策

## 10.1 摘要优先原则

后续系统日常读取优先级应为：

1. archive summary
2. stabilized knowledge
3. validated experience
4. recent reflection
5. raw memory
6. raw logs / session transcript

## 10.2 摘要内容要求

每份摘要至少应保留：

- 核心结论
- 关键原因
- 关键关系
- 适用边界
- 禁止误用场景

## 10.3 归档后的使用方式

- 主程序默认先读摘要
- 只有摘要不足以支撑判断时，才回看原始材料
- 归档不等于遗忘，而是把高频读取入口换成压缩层

## 11. 治理平面与运行平面的分工

## 11.1 运行平面

运行平面由 agent、group、task、session 组成，负责：

- 日常对话
- 任务执行
- 局部记忆形成
- 局部反思输入产生

## 11.2 治理平面

治理平面由 main 所代表，负责：

- 修复
- 审核
- 全局配置
- 上行筛选
- 规则确认

## 11.3 分工边界

- 运行平面负责产出材料
- 治理平面负责筛选高层成果
- 治理平面不直接接管组内日常对话
- 运行平面不直接改写主程序规则

## 12. 0C 的禁入项

以下做法视为阶段 0C 明确禁入：

- 把 session transcript 直接当作长期知识库
- 把 reflection 直接当作 knowledge
- 把单次经验直接上行为主程序规则
- 把 main 设计成会直接进群对话的治理角色
- 用“上下文太长”作为不做归档机制的理由
- 让摘要层失去适用边界和误用提示

## 13. 阶段 0C 的结论清单

本阶段先固定以下结论：

1. **原始材料、记忆、反思、经验、知识、上行知识必须分层**
2. **升级越往上，验证越严格，数量越少**
3. **任何高层产物都不能直接来自原始日志**
4. **超过 500 行的日志、记忆、反思必须触发归档与摘要**
5. **新任务开始前应优先读取摘要、反思和知识层**
6. **main 只负责治理审核，不直接参与组内日常对话治理**
7. **upward knowledge 只能通过治理平面审核后进入更高层**
8. **后续实现应把反思、归档、上行机制做成独立能力，而不是硬编码进主循环**

## 14. 对后续阶段的直接约束

### 对阶段 1 的约束

- agent 身份必须足够稳定，才能承接完整生命周期归属

### 对阶段 2 的约束

- 记忆系统必须原生支持分层、归属、升级、归档和摘要

### 对阶段 3 的约束

- EvoMap 式进化必须以 reflection 和 experience 为中间层，不能跳层

### 对阶段 4 的约束

- Private Moltbook 的社会化交互必须能作为原始材料进入这条生命周期

### 对阶段 5 的约束

- 向上知识反哺只能消费稳定后的高层产物，不能消费组内原始材料
