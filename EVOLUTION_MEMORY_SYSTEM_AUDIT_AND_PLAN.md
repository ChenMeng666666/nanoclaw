# 进化系统与记忆系统审计报告与开发拆解（Phase + Task）

## 1. 审计范围与结论

- 审计对象：`EvolutionManager`、`MainEvolutionApplier`、Runtime API（evolution/memory）、`MemoryManager`、`ContextEngine` 以及相关配置与测试。
- 目标：识别“实现不对”“架构不对”与“可优化”项，并形成可执行的分阶段任务清单。
- 结论：当前系统具备完整的基础能力骨架，但存在**两类关键问题**：
  1) 进化闭环中的晋升逻辑与调用路径不一致，导致关键路径几乎无法产出 Capsule。  
  2) 记忆主链路在文档决策与运行时实现之间存在双轨偏差，长期会造成治理与观测分裂。

## 2. 外部基线现状（evolver + wiki）

- 当前提供的两个参考链接均不可访问（404 / not found），因此无法做逐条协议对照。
- 影响：本次审计采用“仓库内实现一致性 + 可验证行为 + 架构闭环完整性”作为评估主基线。
- 建议：在 P1 阶段补齐可访问、可版本化的外部规范镜像（或本地冻结规范），避免后续演进失去基准。

## 3. 当前实现亮点（保留项）

- 进化系统已具备：经验提交、信号去重、自动审核、GDI 评分、Capsule 与生态指标等能力入口。
- 记忆系统已具备：混合检索（query expansion + BM25 + 向量 + rerank + explain）、生命周期迁移、运行时 API、发布控制与回滚接口。
- 安全基础较好：验证命令白名单与危险操作符拦截机制已存在。

## 4. 关键问题清单（按优先级）

### P0（必须立即修正）

1. **Capsule 晋升调用与判定冲突（实现错误）**
   - 现状：`MainEvolutionApplier` 调用创建 Capsule 时固定传 `outcome.status = 'partial'`。
   - 但 `EvolutionManager.shouldPromoteToCapsule` 要求 `outcomeStatus === 'success'` 才可晋升。
   - 结果：主链路几乎稳定失败，形成“有流程、无产出”的假闭环。

2. **Capsule 冷启动门槛设计不合理（架构/策略错误）**
   - 现状：晋升使用 `capsules.length` 作为 successCount，阈值默认 `minSuccessCount=3`。
   - 风险：首个 Capsule 难以出现，系统可能长期停留在“永不晋升”状态。

### P1（高优优化）

3. **进化配置未闭环消费（实现不完整）**
   - `duplicateThreshold`、`gdiPromotionThreshold`、`metricsSnapshotInterval` 已定义但未被主流程有效消费。
   - 风险：配置与行为脱节，运维调参无效，错误感知延迟。

4. **进化 API 输入约束弱于记忆 API（安全与稳定性风险）**
   - evolution 路由当前缺少与 memory 路由同等级的边界校验（query/limit/tags/rating 类型与范围）。
   - 风险：异常输入、资源浪费、行为不可预测。

### P2（中期架构治理）

5. **记忆系统“唯一主链路”与运行时双轨并存（架构偏差）**
   - ADR 指向 `MemoryManager` 作为当前唯一主链路。
   - 但主消息处理仍直接经过 `ContextEngine` ingest/assemble。
   - 风险：指标口径分裂、治理策略分叉、问题排查成本持续上升。

6. **进化关键流程集成测试不足（质量风险）**
   - 当前偏重 `main-evolution-applier` 单测，缺少 `EvolutionManager` 核心闭环集成级验证。
   - 风险：逻辑回归无法被及时发现，线上才暴露。

### P3（持续优化）

7. **进化/记忆跨系统观测缺统一看板**
   - 缺少“信号→Gene→Capsule→反馈→GDI”与“L1→L2→L3→召回质量”的统一视图。

8. **规范管理缺失**
   - 外部基线不可用时未见“冻结版规范 + 本地镜像 + 版本差异记录”的制度化流程。

## 5. 建议的目标架构（方向）

### 5.1 进化系统目标
- 形成“可晋升、可观测、可回归”的稳定闭环：Signal → Gene → Validation → Capsule → Feedback → GDI。
- 所有阈值统一走配置消费，避免硬编码分叉。
- 晋升策略支持冷启动，避免首个 Capsule 永远无法产生。

### 5.2 记忆系统目标
- 在一个阶段内明确唯一主链路：
  - 路线 A：短期保持 `MemoryManager` 主链路，`ContextEngine` 仅消费不编排；
  - 路线 B：切换到 `ContextEngine` 统一编排，`MemoryManager` 下沉为存储与检索基础服务。
- 不论选 A/B，必须统一可观测指标与调试入口，禁止双口径长期并行。

## 8. 验收标准（建议）

- P0 验收：能在测试环境稳定产出首个 Capsule，且失败路径有明确错误码/日志。
- P1 验收：阈值调整可在不改代码情况下改变行为，API 异常输入全部被拒绝并返回规范错误。
- P2 验收：仅保留一个生产主链路，观测口径统一，回归测试通过。
- P3 验收：完成制度化治理，具备季度可审计输出。

## 9. 风险与回滚策略（执行时参考）

- 风险最高项：P0 晋升逻辑改造可能影响既有数据口径与历史判断。
- 保护措施：按“特性开关 + 小流量验证 + 只读评估模式”分步上线。
- 回滚原则：保持旧判定路径可配置回退，避免一次性切断。

## 10. 建议执行顺序

- 先 P0 再 P1：先让系统“能正确跑起来”，再让系统“可配置可治理”。
- P2 必须在 P1 稳定后执行：避免链路收敛期间再叠加行为不确定性。
- P3 与业务迭代并行推进：制度建设不阻塞功能交付，但必须持续落地。
