## 2. 影响分析

### 2.1 影响架构变更判断的关键要素

- 关键要素 1：迁移链路可观测性不足（DDL 回填吞错、部分迁移无强失败策略）。
- 关键要素 2：跨域依赖未完全收敛（会放大改动传导范围）。
- 关键要素 3：接口契约文档存在漂移（代码、OpenAPI、运行时文档不完全同步）。
- 关键要素 4：兼容 facade 尚未收口（新老路径并存带来决策噪声）。

### 2.2 现有功能在新架构下可行性

- 高可行（可直接迁移）：配置聚合、通道注册、Skill apply 主流程（边界清晰）。
- 中可行（需先解耦）：message pipeline、runtime route 聚合、security 应用服务（跨域强依赖）。
- 低可行（需先治理）：learning scheduler 与 collaboration 旧链路（文档已标注为迁移待办重点）。  
  证据：`docs/DDD_MIGRATION_PLAN.md#L254-L317`

### 2.3 数据迁移与接口变更复杂度

- 数据迁移复杂度：高
  - 启动时大量 `ALTER TABLE` + 回填 + 宽松异常处理。  
    证据：`src/db.ts#L28-L230`
  - JSON 状态迁移采用文件重命名与局部告警，缺少统一回滚控制。  
    证据：`src/infrastructure/persistence/sqlite/migrations/json-state.ts#L13-L69`
  - DB 路径历史双轨（`store/messages.db` 与 `nanoclaw.db`）需统一迁移目标。  
    证据：`src/db.ts#L255-L262`、`scripts/migrate-add-routing-bindings.ts#L14-L21`
- 接口变更复杂度：中高
  - 路由集中有利于治理，但接口面较大且历史兼容层仍在。  
    证据：`src/contexts/runtime/interfaces/http/runtime-api-router.ts#L49-L111`
  - Runtime API 文档与 OpenAPI 覆盖范围不一致。  
    证据：`docs/RUNTIME_API.md#L20-L190`、`docs/openapi.yaml#L604-L634`、`docs/openapi.yaml#L1387-L1457`

### 2.4 技术债与风险点（按优先级）

- P0：迁移治理缺口（Phase 9 未完成的结构性问题）  
- P0：跨层反向依赖（domain 直连 db/application）  
- P1：历史测试波动与失败集中区  
- P1：关键链路硬失败（`process.exit`）导致发布韧性不足  
- P2：兼容 facade 长期滞留带来的维护成本

## 5. 风险缓解策略与回滚方案

### 5.1 风险缓解策略

- 技术策略：
  - 所有结构性改动先做“兼容适配层 + 灰度开关”，再切主路径
  - 数据迁移统一走版本化任务，禁止继续扩散启动时隐式迁移
  - 接口变更执行“双文档一致性门禁”（代码路由 vs OpenAPI vs Runtime API 文档）
- 流程策略：
  - 先拆高耦合点再引入新能力
  - 每阶段结束必须通过：类型检查 + 核心回归 + 迁移预演
  - 风险项绑定 owner、触发条件、观测指标、回退动作

### 5.2 回滚方案（分层）

- 配置回滚：保留上一版本配置快照，一键恢复并重启服务。
- 接口回滚：在兼容窗口内保留旧 handler 路由开关，异常时切回旧路径。
- 数据回滚：
  - 迁移前创建 DB 快照；
  - 迁移任务按“可逆脚本 + 版本号”执行；
  - 失败触发后回滚至最近稳定快照并执行一致性校验。
- 发布回滚：灰度发布设置 SLO 门限（错误率、延迟、关键任务失败率），超阈值自动回滚。
